# DDD Start #6 — 응용 서비스와 표현 영역

> 최범균 『도메인 주도 개발 시작하기』 6장 정리 노트 (본인 필기).

## Summary

도메인 영역을 잘 구현해도 사용자가 그 기능을 쓰려면 사용자와 도메인을 잇는 매개체가 필요하며, 그 역할을 표현 영역과 응용 영역이 맡는다. 표현 영역은 사용자 요청을 해석해 알맞은 응용 서비스를 호출하고 결과를 사용자용 형식으로 변환하며, 응용 서비스는 표현 영역과 도메인 영역을 잇는 창구로서 리포지터리에서 도메인 객체를 구해 기능 실행 흐름만 제어한다. 핵심 원칙은 **응용 서비스가 도메인 로직을 갖지 않고**(도메인 영역에 모아 응집도↑·중복↓) **표현 영역에 의존하지 않는 것**(`HttpServletRequest` 등 배제)이다. 이 장은 응용 서비스의 크기·인터페이스 필요성·파라미터/리턴 설계·트랜잭션 처리, 표현 영역의 세 책임, 값 검증 위치, 권한 검사(표현/응용/도메인 3계층), 그리고 조회 전용 기능에서는 응용 서비스를 생략할 수 있다는 점을 다룬다.

## Key Claims

1. **역할 분리.** 표현 영역은 요청 해석과 결과 변환을, 응용 서비스는 실제 기능 제공을 담당한다. 응용 서비스는 사용자가 웹/REST/소켓 중 무엇을 쓰는지 몰라야 한다(표현 영역 비의존).
2. **응용 서비스 흐름은 단순하다.** 기존 애그리거트 사용: 조회→(없으면 예외)→도메인 기능 실행→리턴. 새 애그리거트 생성: 입력 검증→생성→저장→리턴. 응용 서비스가 복잡해지면 도메인 로직이 새어 들어갔을 가능성이 높다.
3. **도메인 로직을 응용 서비스에 넣지 않기.** 예: 암호 변경 시 기존 암호 확인은 도메인 규칙이므로 애그리거트가 담당해야 한다. 응용 서비스가 직접 비교·세팅하면 (1) 데이터와 로직이 흩어져 응집도가 낮아지고 (2) 여러 기능에서 중복 구현된다(탈퇴 시에도 암호 확인 등).
4. **응용 서비스 크기**는 도메인당 한 클래스에 몰기 vs 기능별 분리 두 방식이 있다. 저자는 구분되는 기능별 서비스 분리를 선호하며, 중복 로직은 공통 헬퍼로 뺀다.
5. **인터페이스는 대개 불필요(YAGNI).** 구현이 여럿이거나 런타임 교체가 필요할 때만 유용한데 응용 서비스는 둘 다 드물다. Mockito가 클래스도 대역으로 만들 수 있어 단위 테스트에도 인터페이스가 꼭 필요하지 않다.
6. **파라미터/리턴.** 값이 둘 이상이면 별도 요청 클래스로 묶는 편이 편리하다. 표현 영역에 필요한 데이터만 리턴하고, **애그리거트 자체를 리턴하는 것은 권장하지 않는다**(표현 영역에서 도메인 기능 실행 여지가 생겨 로직이 분산됨).
7. **표현 영역 비의존.** `HttpServletRequest`/`HttpSession`을 응용 서비스에 넘기면 단독 테스트가 어렵고, 표현 영역 변경이 전파되며, 응용 서비스가 표현 영역 역할(세션 조작)까지 하게 된다.
8. **트랜잭션은 응용 서비스의 몫.** 상태 변경을 트랜잭션으로 묶어 화면 결과와 DB 상태의 불일치를 막는다. 스프링 `@Transactional`은 `RuntimeException`이면 롤백, 아니면 커밋한다.
9. **표현 영역의 세 책임**: 화면 흐름 제어, 요청→응용 서비스 전달 및 결과 제공, 세션 관리(권한 검사와 연결).
10. **값 검증.** 원칙적으로 응용 서비스에서 처리한다. 다만 첫 오류에서 예외를 던지면 사용자가 모든 오류를 한 번에 못 봐 UX가 나빠지므로, 오류를 목록으로 모아 단일 예외로 던져 `BindingResult`에 담거나, 표현 영역에서 필수/형식/범위를 먼저 검사하고 응용 서비스는 논리 검증(중복·존재 여부)을 맡는 분담도 가능하다. 저자는 지금은 필수 값 검증과 논리 검증을 모두 응용 서비스에서 하는 편이라고 한다.
11. **권한 검사 3계층**: 표현 영역(인증 여부, 서블릿 필터/스프링 시큐리티), 응용 서비스 메서드(스프링 시큐리티 AOP `@PreAuthorize`), 도메인 객체 수준(작성자 본인 확인 등, 애그리거트 조회 후 권한 검사 서비스 호출). 프레임워크 이해가 부족하면 확장보다 직접 구현이 유지보수에 유리할 수 있다.
12. **조회 전용 기능은 응용 서비스를 생략할 수 있다.** 조회 전용 DAO를 호출해 그대로 반환할 뿐이면 트랜잭션도 불필요하므로 표현 영역에서 DAO/리포지터리를 바로 써도 된다(11장 CQRS에서 재론).

## Caveats

- ⚠️ raw 노트에는 Application Service·Controller 코드 스크린샷이 포함돼 있으나 이 정리에서는 생략했다.
- ⚠️ **6.3.1의 인용문("중복 로직을 Service 아래에 넣어 해결… 제민님의 『지속 가능한 소프트웨어 아키텍처』의 Implement Layer")은 책 내용이 아니라 사용자가 덧붙인 해석·외부 참조다.**
- ⚠️ 값 검증 엄격성·권한 검사 프레임워크 도입 여부는 "의견이 갈린다"고 명시된 부분으로, 단정적 규범이 아니다.
