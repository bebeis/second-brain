# DDD Start #11 — CQRS

> 최범균 『도메인 주도 개발 시작하기』 11장 정리 노트 (본인 필기).

---

## Summary

조회 화면은 여러 애그리거트에 흩어진 데이터를 모아야 하는 경우가 많은데, 상태 변경과 조회에 **같은 도메인 모델**을 쓰면 조회 성능을 맞추느라 모델이 불필요하게 복잡해진다. 객체 지향 도메인 모델과 ORM은 상태 변경에는 적합하지만 조회 출력에는 고려할 것이 많기 때문이다. 이를 푸는 패턴이 **CQRS(Command Query Responsibility Segregation)** — 상태를 변경하는 **명령 모델**과 상태를 제공하는 **조회 모델**을 분리한다. 두 모델은 설계 초점·기술·저장소가 달라도 되고(명령=JPA/RDBMS, 조회=MyBatis/NoSQL), 저장소를 분리하면 10장의 이벤트로 동기화한다. 웹 서비스는 조회 요청이 훨씬 많아 캐싱·쿼리 최적화가 사실상 CQRS와 비슷한 효과를 내며, 도입은 코드·기술이 늘어나는 단점과 도메인 집중·조회 성능이라는 장점을 비교해 결정한다.

---

## Key Claims

1. **단일 모델의 단점**: 조회에 여러 애그리거트 데이터가 필요할 때 식별자 참조는 한 번의 SELECT로 가져오기 어렵고, 객체 참조는 화면마다 로딩 전략이 달라지고 네이티브 쿼리가 필요할 수 있다. 근본 원인은 상태 변경과 조회에 같은 모델을 쓰기 때문이다.
2. **CQRS**: 명령 모델(상태 변경)과 조회 모델(상태 제공)을 분리하는 패턴. 도메인이 복잡할수록 명령/조회가 다루는 데이터 범위가 달라 적합하다.
3. **데이터 범위 불일치**: 상태 변경은 주로 한 애그리거트, 조회는 여러 애그리거트를 함께 필요로 한다. 범위가 다르므로 한 모델로 둘 다 하면 복잡해진다.
4. **모델 설계 초점**: 명령 모델은 상태 변경 도메인 로직에, 조회 모델은 화면 데이터를 빠르게 읽는 데 초점. 같은 기능이라도 조회 모델은 화면에 필요한 데이터만 담은 요약 타입 하나로 구성할 수 있다.
5. **기술 자유**: 명령=JPA, 조회=MyBatis처럼 각 모델에 맞는 기술을 고를 수 있고, 반드시 다른 기술일 필요는 없다(5장 JPQL 동적 인스턴스, 하이버네이트 `@Subselect`도 조회 모델).
6. **저장소 분리와 동기화**: 명령=RDBMS, 조회=메모리 기반 NoSQL도 가능. 동기화는 10장 이벤트로 — 즉시 반영은 동기 이벤트+글로벌 트랜잭션(응답 속도·처리량 저하), 특정 시간 내면 비동기(성능 저하 완화).
7. **조회 모델엔 응용 서비스가 없을 수도 있다**: 단순 조회는 컨트롤러가 DAO를 바로 실행해도 된다(6장 조회 전용 기능과 연결).
8. **웹과 CQRS**: 웹은 변경보다 조회 요청이 훨씬 많다. 쿼리 최적화·화면 형태 캐싱·조회 전용 저장소는 결국 조회 전용 모델을 쓰는 것과 같아, 대규모 트래픽 서비스는 명시하지 않아도 CQRS 비슷한 구조가 된다.
9. **장점**: 명령 모델이 도메인 로직에 집중(조회 성능용 코드가 안 섞임), 조회 성능 향상 자유(캐시·특화 쿼리·전용 저장소).
10. **단점**: 코드가 늘고, 두 모델·저장소·동기화용 메시징 등 더 많은 기술이 필요할 수 있다. 단순한 도메인은 유지 비용만 커지므로 장단점을 비교해 도입한다.

---

## Caveats

- ⚠️ raw 노트에는 단일 모델 문제와 CQRS 구조, 명령/조회 분리 그림이 있으나 정리에서는 생략했다.
- ⚠️ 원문은 CQRS 이해를 돕는 참고 영상 안내가 책에 함께 제시돼 있다고만 언급하고 링크는 남기지 않았다.
