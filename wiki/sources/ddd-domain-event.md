# DDD Start #10 — 이벤트

> 최범균 『도메인 주도 개발 시작하기』 10장 정리 노트 (본인 필기).

---

## Summary

한 도메인 기능 뒤에 외부 시스템 연동을 동기로 직접 호출하면 트랜잭션 처리 애매함과 성능 전이 문제가 생기고, 그 원인은 두 바운디드 컨텍스트의 **강결합**이다. 이를 낮추는 방법이 **도메인 이벤트**, 특히 **비동기 이벤트**다. 이벤트는 "과거에 벌어진 일"이며 보통 상태 변경을 뜻하고, (이벤트, 생성 주체, 디스패처/퍼블리셔, 핸들러/구독자) 네 구성요소로 구현된다. 스프링에서는 `ApplicationEventPublisher`와 `@EventListener`로 구현하지만, 핸들러가 동기로 실행되면 외부 서비스의 성능·트랜잭션 영향이 그대로 남는다. 비동기 처리 방법은 로컬 핸들러(`@Async`)·메시지 큐·이벤트 저장소+포워더·이벤트 저장소+API 네 가지이며, `@TransactionalEventListener(AFTER_COMMIT)`로 트랜잭션 성공 후에만 핸들러를 실행해 경우의 수를 줄인다. 손실·순서·재처리·멱등성도 함께 고려한다.

---

## Key Claims

1. **강결합 문제**: 외부 시스템을 동기로 직접 호출하면 (1) 외부 예외 시 내 트랜잭션 롤백 여부가 애매하고 (2) 외부가 느려지면 내 기능도 느려진다. 도메인 객체에 외부 서비스가 섞이면 변경 전이도 커진다.
2. **이벤트 = 과거의 사건**: 보통 상태 변경을 의미한다. 요구사항의 "~할 때", "~가 발생하면"은 상태 변경과 관련된 경우가 많아 이벤트로 표현할 수 있다. 이름은 **과거 시제**(`Changed`, `Canceled`, `Created`).
3. **네 구성요소**: 이벤트, 이벤트 생성 주체(엔티티·밸류·도메인 서비스), 디스패처/퍼블리셔, 핸들러/구독자.
4. **이벤트 구성**: 종류(클래스 이름), 발생 시간, 처리에 필요한 추가 데이터. 핸들러가 필요한 최소 데이터를 담되 무관한 데이터는 넣지 않는다. 부족하면 핸들러가 다시 조회해야 한다.
5. **용도**: (1) 후처리 **트리거**(주문 취소→환불), (2) 시스템 간 **데이터 동기화**(배송지 변경→외부 배송 서비스).
6. **장점**: 서로 다른 도메인 로직의 섞임을 막는다. `Order.cancel()`이 환불 서비스를 직접 부르지 않고 `OrderCanceledEvent`만 발생시켜 **주문→결제 의존을 제거**하고, 이메일 통지 등 확장은 핸들러 추가만으로 된다.
7. **스프링 구현**: 이벤트 클래스(상위 타입 불필요, 공통 속성 있으면 `Event` 추상 클래스로 `timestamp`), `ApplicationEventPublisher`(정적 헬퍼 `Events.raise()`가 `publishEvent()` 호출), `@EventListener(타입.class)` 핸들러. 기본적으로 응용 서비스와 핸들러가 **같은 트랜잭션**에서 실행된다.
8. **동기 처리의 한계**: 핸들러가 동기면 외부 서비스가 느릴 때 원 기능도 느려지고 트랜잭션 롤백 고민이 남는다. → 비동기 처리 또는 트랜잭션 연계 필요.
9. **비동기가 맞는 요구**: "A 하면 B"는 실제로 "A 하면 최대 언제까지 B"인 경우가 많다(가입 후 인증 메일, 취소 후 결제 취소). 이런 경우 이벤트를 비동기로 처리한다.
10. **비동기 4방법**: (1) 로컬 핸들러 비동기(`@EnableAsync`+`@Async`), (2) 메시지 큐(카프카/래빗MQ; 도메인 반영과 큐 저장을 묶으려면 글로벌 트랜잭션 — 래빗MQ 지원, 카프카 미지원·고성능), (3) 이벤트 저장소+포워더(같은 DB 로컬 트랜잭션, 재처리 가능, 포워더가 처리 위치 추적), (4) 이벤트 저장소+API(외부 핸들러가 API로 이벤트를 가져가고 스스로 offset 추적).
11. **이벤트 저장소 구현**: `EventEntry`(id·타입·payload(JSON)·발생시간), `EventStore`(불변이라 `save`/`get(offset,limit)`만), `EventApi`(`/api/events?offset=&limit=`), `EventStoreHandler`(`Event` 상위 타입만 저장). 클라이언트는 `lastOffset`을 저장해 중복 처리를 막는다.
12. **자동 증가 칼럼 주의**: 증가 값은 insert 시점, 실제 반영은 커밋 시점이라 id 11 미커밋·id 12 커밋 사이 조회 시 구멍이 생긴다. **지연 조회** 등을 고려한다.
13. **추가 고려**: (1) 이벤트 소스 정보 저장 여부, (2) 포워더 재시도 횟수 제한(예 3회 후 스킵·별도 실패 저장), (3) 이벤트 손실(저장소 방식은 트랜잭션으로 보장, 로컬 비동기는 유실 가능), (4) 순서(저장소는 발생 순서 보장, 메시징은 기술에 따라 달라짐), (5) 재처리(마지막 순번 기억 또는 **멱등** 핸들러).
14. **트랜잭션 연계**: `@TransactionalEventListener(phase = AFTER_COMMIT)`로 커밋 성공 뒤에만 핸들러를 실행(롤백 시 미실행). 이벤트 저장소를 DB로 쓰는 방식도 발생·저장을 한 트랜잭션으로 묶어 같은 효과를 낸다.

---

## Caveats

- ⚠️ raw 노트에는 이벤트 구성요소 다이어그램, `Events`/`ApplicationEventPublisher`/`EventStore`/`Forwarder` 코드와 흐름 그림이 다수 있으나 정리에서는 개념·흐름만 남겼다.
- ⚠️ **[본인 주석]** 원문에 사용자가 덧붙인 코멘트: `@TransactionalEventListener(AFTER_COMMIT)`를 붙인다고 "모든 게 해결되는 게 아님" — 트랜잭션 성공 후에도 **이벤트 처리 실패**는 여전히 남아 재시도·실패 저장·수동 처리·멱등 처리 중에서 선택해야 한다는 취지다.
