# DDD Start #3 — 애그리거트

> 최범균 『도메인 주도 개발 시작하기』 3장 정리 노트 (본인 필기).

## Summary

애그리거트는 관련 엔티티·밸류를 하나의 군집으로 묶어 상위 수준에서 도메인을 보게 해주는 개념으로, 일관성을 관리하는 기준이자 경계다. 애그리거트에는 루트 엔티티가 있고, 루트는 구성 객체 전체가 일관된 상태를 유지하도록 도메인 기능을 제공한다. 외부에서 내부 객체를 직접 변경하지 못하게 public set을 없애고 밸류를 불변으로 만든다. 트랜잭션 범위는 작을수록 좋아 한 트랜잭션에서 가급적 하나의 애그리거트만 수정하며, 여럿을 함께 바꿔야 하면 응용 서비스나 도메인 이벤트를 쓴다. 리포지터리는 애그리거트 단위로 만들고, 다른 애그리거트는 객체가 아니라 **ID로 참조**해 결합도·성능·확장 문제를 줄인다. ID 참조는 N+1 조회 문제를 낳을 수 있어 조회 전용 쿼리로 보완하고, 애그리거트 간 1-N/M-N은 컬렉션 대신 N쪽 ID 참조로 구현한다. 끝으로 한 애그리거트의 데이터로 다른 애그리거트를 만들 때 애그리거트를 팩토리로 쓰는 방법을 다룬다.

## Key Claims

1. **애그리거트는 관련 객체를 군집으로 묶어 애그리거트 간 관계로 도메인을 이해하게 한다.** 복잡도가 낮아지고 확장·변경 부담이 준다.
2. **애그리거트는 일관성·라이프사이클·경계의 기준이다.** 같은 애그리거트 객체는 함께 생성·제거되고, 한 애그리거트는 다른 애그리거트의 상태(회원 비밀번호·상품 가격 등)를 변경하지 않는다.
3. **경계는 도메인 규칙·요구사항 기준으로 정한다.** "A가 B를 가진다"는 관계만으로 묶지 않는다. 상품과 리뷰는 함께 보여도 생성 시점·변경 주체·라이프사이클이 달라 다른 애그리거트다. 실제 애그리거트 크기는 생각보다 작고, 엔티티 하나짜리도 흔하다.
4. **애그리거트 루트가 일관성을 책임진다.** 상품 수량이 바뀌면 OrderLine.quantity와 Order.totalAmounts가 함께 바뀌어야 규칙이 안 깨진다. `order.getShippingInfo().setAddress(...)`처럼 루트를 우회하면 안 된다.
5. **일관성 보호 습관 두 가지:** (1) 단순 필드 변경용 public set을 만들지 않는다, (2) 밸류를 불변으로 구현한다. 내부 객체를 노출하더라도 변경 기능은 노출하지 않는다(불변화 또는 접근 범위 제한).
6. **트랜잭션 범위는 작을수록 좋다 → 한 트랜잭션에서 하나의 애그리거트만 수정한다.** 여러 개를 바꿔야 하면 응용 서비스에서 각각 조회해 기능을 호출하거나 **도메인 이벤트**(10장)를 쓴다. 팀 표준·기술 제약·UI 편의로 예외가 있을 수 있다.
7. **리포지터리는 애그리거트 단위로 만든다.** OrderLine용 리포지터리는 따로 두지 않고 루트 기준 `OrderRepository`만 둔다. 기본 기능은 `save`, `findById`. 저장·조회·변경은 애그리거트 전체를 원자적으로 다룬다(RDBMS는 트랜잭션, NoSQL은 한 문서).
8. **다른 애그리거트는 ID로 참조한다.** 객체 참조는 (1) 결합도 상승 (2) 성능 고민 증가 (3) 저장소 분리 시 확장 어려움을 낳는다. ID 참조(Orderer가 MemberId 보관)는 경계를 명확히 하고 응집도를 높이며 저장소 확장에 유리하다.
9. **ID 참조는 N+1 조회 문제를 부른다 → 조회 전용 쿼리/DAO로 조인해 한 번에 가져온다.** 저장소가 다르면 캐시나 조회 전용 저장소를 따로 구성한다.
10. **애그리거트 간 1-N/M-N은 컬렉션 대신 ID 참조로 구현한다.** 1쪽이 N쪽 컬렉션을 들면 페이징 성능 문제가 생기므로 N쪽이 상대 ID를 갖고 조회한다. M-N은 보통 단방향(한쪽이 상대 ID 목록)으로 충분하다.
11. **애그리거트를 팩토리로 쓸 수 있다.** "차단된 상점은 상품 등록 불가" 같은 규칙이 응용 서비스에 노출되지 않도록, Store에 `createProduct()` 팩토리 메서드를 두고 내부에서 차단 여부를 검사한다. 도메인 로직의 응집도가 높아진다. 정보가 너무 많으면 별도 팩토리에 위임하되 핵심 도메인 로직은 애그리거트에 남긴다.

## Caveats

- ⚠️ 원본 노트의 애그리거트/루트 다이어그램, ID 참조·저장소 분리 도식, 팩토리 코드 이미지는 정리에서 생략했다. 정확한 도식은 [raw 원문](../../raw/ddd-start/도메인%20주도%20개발%20시작하기%20%233%20애그리거트.md) 참조.
- ⚠️ 도메인 이벤트를 통한 다중 애그리거트 변경은 이 장에서 언급만 하고 상세는 10장에서 다룬다.
