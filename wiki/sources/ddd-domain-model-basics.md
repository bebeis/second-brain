# DDD Start #1 — 도메인 모델 시작하기

> 최범균 『도메인 주도 개발 시작하기』 1장 정리 노트 (본인 필기).

## Summary

도메인은 소프트웨어로 해결할 문제 영역이며, 하나의 도메인은 카탈로그·주문·결제·배송 같은 여러 하위 도메인으로 나뉜다. 하위 도메인 구성은 비즈니스 상황에 따라 달라지고, 모든 기능을 직접 구현하지 않고 결제 PG·배송 추적처럼 외부 시스템과 연동하기도 한다. 요구사항을 정확히 이해하려면 개발자와 도메인 전문가가 직접 소통해야 하며, 전문가의 표면적 요구를 그대로 받지 말고 진짜 문제를 확인해야 한다. 도메인 모델은 도메인을 개념적으로 표현한 것이고, 도메인 모델 패턴은 핵심 업무 규칙을 도메인 객체 안에 구현하는 것을 뜻한다. 도메인 모델은 크게 식별자를 가진 **엔티티**와 개념적으로 완전한 하나의 값을 표현하는 **밸류**로 나뉜다. 마지막으로 전문가·관계자·개발자가 함께 쓰는 공통 언어인 **유비쿼터스 언어**를 대화·문서·코드에 일관되게 반영해야 한다고 강조한다.

## Key Claims

1. **도메인은 여러 하위 도메인으로 나뉘고, 그 구성은 고정돼 있지 않다.** 비즈니스 상황(대형 장비 판매 vs 일반 소비자 대상 등)에 따라 필요한 하위 도메인이 달라진다.
2. **좋은 요구사항 이해는 도메인 전문가와의 직접 대화에서 나온다.** 중간 전달자가 많을수록 정보가 왜곡·누락되고, 잘못된 요구사항은 잘못된 제품을 낳는다. 전문가도 기존 업무 방식에 맞춘 해결책을 요구할 수 있으므로 진짜 목적을 확인해야 한다.
3. **도메인 모델은 주요 데이터와 기능을 함께 표현한다.** 반드시 객체일 필요는 없고(상태 다이어그램·그래프·수식 등), 중요한 것은 표현 방식이 아니라 도메인 이해에 도움이 되는가이다.
4. **개념 모델과 구현 모델은 다르다.** 개념 모델은 DB·트랜잭션·성능을 고려하지 않으므로 구현 기술에 맞는 구현 모델로 전환해야 하며, 처음부터 완벽할 수 없으므로 점진적으로 발전시킨다.
5. **하위 도메인마다 같은 용어가 다른 의미를 가질 수 있다.** 카탈로그의 "상품"과 배송의 "상품"은 다르므로 하위 도메인마다 별도 모델을 만든다.
6. **도메인 모델 패턴은 업무 규칙을 도메인 객체 안에 둔다.** "출고 전에는 배송지 변경 가능" 같은 규칙을 Order·OrderState에 구현해, 규칙 변경 시 변경 범위를 줄인다.
7. **엔티티는 식별자로 구분되고, 식별자는 생성부터 삭제까지 유지된다.** 식별자가 같으면 같은 엔티티이며 보통 식별자 기준으로 equals/hashCode를 구현한다. 식별자 생성은 규칙 조합·UUID/Nano ID·사용자 입력·DB 시퀀스 등으로 한다.
8. **밸류는 개념적으로 완전한 하나를 표현한다.** (Receiver, Address, Money 등) 필드가 하나여도 되고, 관련 기능(Money.add 등)을 가질 수 있으며, **불변으로 구현**하고 모든 속성이 같은지로 비교한다.
9. **엔티티 식별자도 밸류 타입으로 만들면 의미가 명확하다.** String 대신 OrderNo, MemberId를 쓴다.
10. **도메인 모델에 public set을 두지 않는다.** setShippingInfo보다 changeShippingInfo가 의도를 드러내고, set은 객체를 불완전한 상태로 만든다. 생성 시점에 필요한 값을 생성자로 모두 전달하고 제약을 검사한다. 내부용 private set은 허용.
11. **유비쿼터스 언어를 코드에 반영한다.** STEP1/STEP2 대신 PAYMENT_WAITING·PREPARING처럼 도메인 용어를 쓰면 번역 비용과 버그가 줄고, 이해가 깊어지면 더 나은 용어로 코드·문서를 갱신한다.

## Caveats

- ⚠️ 원본 노트의 도식 이미지(도메인 구성도, 엔티티/밸류 클래스 다이어그램, Money 코드 등)는 이 정리에서 생략했다. 정확한 도식은 [raw 원문](../../raw/ddd-start/도메인%20주도%20개발%20시작하기%20%231%20도메인%20모델%20시작하기.md) 참조.
- ⚠️ 식별자 생성 방식(UUID/Nano ID/시퀀스 등)의 구체 선택 기준은 노트에 예시 수준으로만 있어, 세부 트레이드오프는 원문 밖 판단이 필요하다.
