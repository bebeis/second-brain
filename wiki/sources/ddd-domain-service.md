# DDD Start #7 — 도메인 서비스 (여러 애그리거트가 필요한 기능)

> 최범균 『도메인 주도 개발 시작하기』 7장 정리 노트 (본인 필기).

## Summary

결제 금액 계산처럼 상품 가격·구매 개수·쿠폰 할인·회원 등급 할인 등 **여러 애그리거트에 흩어진 정보**가 필요한 기능은 특정 애그리거트에 넣기 애매하다. 억지로 한 애그리거트에 넣으면 책임 범위를 넘고, 외부 의존이 늘며, 무엇보다 애그리거트 바깥의 도메인 개념이 내부에 숨어 명시적으로 드러나지 않는다. 이를 푸는 방법이 **도메인 서비스**로, 도메인 영역에 위치해 도메인 로직을 표현하되 **상태 없이 로직만** 구현하고 필요한 상태는 파라미터로 전달받는다. 주 사용처는 (1) 여러 애그리거트가 필요하거나 복잡한 계산 로직, (2) 외부 시스템 연동이 필요한 도메인 로직이다. 응용 서비스가 응용 로직(트랜잭션 등)을 다룬다면 도메인 서비스는 도메인 로직만 다룬다.

## Key Claims

1. **한 애그리거트에 넣기 애매한 도메인 개념은 도메인 서비스로 분리**해 명시적으로 드러낸다. 계산 책임을 억지로 주문 애그리거트에 두면, 주문과 무관한 할인 정책이 생겨도 주문 애그리거트를 수정해야 하는 문제가 생긴다.
2. **도메인 서비스는 상태 없이 로직만 구현한다.** 애그리거트·밸류와 달리 DB 저장 대상이 아니며, 필요한 값(주문 항목·쿠폰·회원 등급 등)을 파라미터로 받아 결과를 반환한다. 예: `DiscountCalculationService`.
3. **사용 주체는 애그리거트 또는 응용 서비스.** 애그리거트 메서드에 도메인 서비스를 **파라미터로 전달**하거나, 응용 서비스가 애그리거트를 준비한 뒤 도메인 서비스에 전달해 실행한다.
4. **도메인 서비스를 애그리거트에 주입(필드)하지 않는다.** 도메인 서비스는 애그리거트의 데이터가 아니고 DB 저장 대상도 아니며, 모든 기능이 그것을 필요로 하지도 않는다. 스프링 DI에 익숙해 필드 주입하고 싶어도 좋은 방법이 아니다 — 메서드 실행 시 파라미터로 전달한다.
5. **반대로 도메인 서비스 메서드에 애그리거트를 전달**하기도 한다. 대표 예가 계좌 이체(출금·입금 두 애그리거트)로, 응용 서비스가 두 애그리거트를 조회해 도메인 서비스에 넘긴다.
6. **응용/도메인 서비스 구분 기준**: 로직이 애그리거트 상태를 변경하는지 / 상태 값을 계산하는지 보면 된다. 계좌 이체(상태 변경)·결제 금액 계산(계산) 둘 다 도메인 로직이며, 한 애그리거트에 넣기 적합하지 않아 도메인 서비스로 구현한다. 트랜잭션 처리는 응용 로직이므로 응용 서비스가 맡는다.
7. **외부 시스템 연동도 도메인 서비스가 될 수 있다.** 설문 생성 권한 확인처럼 역할 관리 시스템과 연동하는 기능도, "역할 시스템과 연동한다"가 아니라 "설문 생성 권한을 확인한다"는 **도메인 로직 관점**으로 `SurveyPermissionChecker` 같은 인터페이스로 표현한다. 구현 클래스는 인프라스트럭처 영역에 둔다.
8. **패키지 위치**는 도메인 영역이다. 도메인 서비스가 많으면 `domain.model`/`domain.service`/`domain.repository`로 하위 패키지를 나눌 수 있다.
9. **인터페이스 분리 기준**: 로직이 고정되지 않았거나 외부 시스템·별도 엔진(룰 엔진 등)으로 구현할 때 도메인 서비스를 인터페이스로 추상화하고 구현체를 인프라에 둔다. 그래야 도메인이 구현 기술에 종속되지 않고 테스트가 쉬워진다.

## Caveats

- ⚠️ raw 노트에는 그림 7.1(도메인/응용 영역 배치)·그림 7.2(인터페이스+룰 엔진 구현체 구조) 등이 언급·삽입되어 있으나 이 정리에서는 이미지·코드를 생략했다.
- ⚠️ 이 장의 도메인 서비스는 [응용 서비스 계층](../topics/ddd-application-service-layer.md)과 짝을 이루는 개념으로, "응용 로직 vs 도메인 로직" 구분이 핵심이다.
