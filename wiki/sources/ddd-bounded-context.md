# DDD Start #9 — 도메인 모델과 바운디드 컨텍스트

> 최범균 『도메인 주도 개발 시작하기』 9장 정리 노트 (본인 필기).

---

## Summary

모든 하위 도메인을 하나의 완전한 모델로 표현하려는 것은 흔한 함정이다. 같은 용어라도 하위 도메인마다 의미가 다르고(카탈로그의 "상품" vs 재고의 "상품") 같은 대상을 다른 용어로 부르기도 하므로, 모델은 특정 문맥 안에서만 완전한 의미를 가진다. 이렇게 경계를 갖는 문맥이 **바운디드 컨텍스트(BC)**다. BC는 논리적으로 하나의 모델을 갖고 주로 용어를 기준으로 구분되며, 이상적으로는 하위 도메인과 1:1이지만 실제로는 조직 구조에 좌우된다. BC는 도메인 모델뿐 아니라 DB·표현 영역까지 포함하는 물리적 시스템이고, 서로 다른 구현 기술·CRUD/CQRS를 섞을 수 있다. 컨텍스트 간에는 통합(직접 REST/간접 메시지 큐)과 관계(고객-공급자, 오픈 호스트 서비스, 안티코럽션 계층, 공유 커널, 독립 방식)가 있으며, 전체 조망을 위해 **컨텍스트 맵**을 그린다.

---

## Key Claims

1. **모델은 문맥 안에서만 완전하다**: 하위 도메인마다 용어·의미가 다르므로 하나의 모델로 전부 표현하면 의미가 약해지고 확장이 어렵다. 모델은 하위 도메인마다 따로, 명확한 경계로 나눈다.
2. **바운디드 컨텍스트의 성질**: 모델 경계를 결정, 논리적으로 한 개의 모델, 주로 용어 기준 구분, 기능을 제공하는 물리적 시스템.
3. **BC와 하위 도메인은 1:1이 이상적**이나 현실은 조직 구조에 따라 갈린다. 소규모 기업은 여러 하위 도메인이 한 BC에 들어가기도 한다. 이때도 **패키지를 구분해 논리적 BC**를 유지해 모델이 섞이지 않게 한다.
4. **같은 대상, 다른 모델**: 사용자는 회원 컨텍스트에서 `Member` 애그리거트 루트, 주문 컨텍스트에서 `Orderer` 밸류. 상품은 카탈로그의 `Product`(Category와 관계)와 재고의 `Product`가 다르다.
5. **BC는 도메인 모델만이 아니다**: DB 테이블, 표현 영역(HTML 또는 REST API)까지 포함한다. 모든 BC를 DDD로 만들 필요는 없고 단순하면 CRUD(서비스-DAO)로 충분하다. 한 BC 안에서 **CQRS**로 명령/조회를 섞을 수도 있다.
6. **BC마다 다른 기술 가능**: 어떤 건 스프링 MVC+JPA, 어떤 건 Netty+MyBatis, 어떤 건 MongoDB. UI가 없어도 되고, 브라우저가 여러 BC의 REST API를 직접 호출하거나 **UI 서버가 파사드**로 조합할 수도 있다.
7. **컨텍스트 통합**: 두 BC의 모델이 다르므로 한쪽 모델을 그대로 쓰면 안 된다. 카탈로그 관점의 도메인 서비스 인터페이스(`ProductRecommendationService`)를 두고, 구현체(`RecSystemClient`)를 인프라에 두어 외부 추천 모델을 카탈로그 모델로 변환한다.
8. **통합 방식**: 직접 통합(REST API 호출)과 간접 통합(메시지 큐). 큐를 누가 제공하느냐에 따라 메시지가 어느 도메인 모델을 따를지 결정된다(**출판/구독 모델**). 큐는 비동기라 발신 측이 처리를 기다리지 않는다.
9. **BC 간 관계**: API를 쓰는 쪽이 제공하는 쪽에 의존한다. **하류(downstream, 사용/고객)**–**상류(upstream, 제공/공급자)**의 고객-공급자 관계이며 상호 협력이 중요하다. 하류가 여럿이면 상류가 **오픈 호스트 서비스**로 단일 API를 공개한다.
10. **안티코럽션 계층**: 상류 모델이 내 도메인 모델을 침범하지 않도록 보호하는 계층. `RecSystemClient`처럼 외부 모델을 내부 모델로 변환한다.
11. **공유 커널**과 **독립 방식**: 두 팀이 하나의 모델을 공유(중복↓, 그러나 밀접한 협력 필요)하거나, 통합하지 않고 수동으로 통합. 수동 통합은 규모가 커지면 한계가 온다.
12. **마이크로서비스와 BC**: 컨텍스트를 마이크로서비스로 구현하면 프로젝트가 분리돼 모델이 자연스럽게 나뉘고 독립 배포·확장이 쉬워 궁합이 좋다.
13. **컨텍스트 맵**: BC 간 관계를 한눈에 보는 지도. 오픈 호스트 서비스·안티코럽션 계층·하위 도메인·조직을 함께 표시하면 좋다. 정해진 규칙 없이 화이트보드로 충분하고, 이해가 깊어지면 함께 갱신한다.

## Caveats

- ⚠️ raw 노트에는 컨텍스트 분리/통합/메시지 구조/컨텍스트 맵 그림이 다수 있으나 정리에서는 개념만 남기고 생략했다.
- ⚠️ `RecSystemClient`가 안티코럽션 계층이자 도메인 서비스 구현체 역할을 겸한다는 서술은 7장(도메인 서비스)의 외부 시스템 연동과 직접 연결된다.
