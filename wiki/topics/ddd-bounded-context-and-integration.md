# 바운디드 컨텍스트와 통합

모델을 어디까지 하나로 볼 것인가, 그리고 서로 다른 모델을 어떻게 잇는가. 9장이 개념·관계·컨텍스트 맵을, 10장이 이벤트를 통한 느슨한 통합을 다룬다.

## 왜 경계가 필요한가
흔한 함정은 **모든 하위 도메인을 하나의 완전한 모델로** 표현하려는 것이다. 같은 "상품"도 카탈로그(정보)와 재고(추적 대상)에서 의미가 다르고, 같은 대상을 하위 도메인마다 다르게 부르기도 한다. 모델은 **특정 문맥 안에서만 완전한 의미**를 가지며, 이 경계를 갖는 문맥이 **바운디드 컨텍스트**다 → [바운디드 컨텍스트](../entities/bounded-context.md). 용어의 문맥 의존성은 1장에서 이미 씨앗이 뿌려졌다([도메인 모델 구성요소](ddd-domain-model-building-blocks.md)). ([#9](../sources/ddd-bounded-context.md))

## 성질과 조직 현실
바운디드 컨텍스트는 모델 경계를 정하고, 논리적으로 한 개의 모델을 가지며, 주로 용어로 구분되고, **기능을 제공하는 물리적 시스템**이다. 이상적으로는 하위 도메인과 1:1이지만 현실은 **조직 구조**에 좌우된다(작은 기업은 여러 하위 도메인이 한 컨텍스트에). 물리적으로 하나여도 **패키지를 구분해 논리적 경계**를 유지해 모델이 섞이지 않게 한다. 모든 컨텍스트를 DDD로 만들 필요는 없고(단순하면 CRUD/서비스-DAO), 한 컨텍스트 안에서 [CQRS](ddd-cqrs-and-read-model.md)로 방식을 섞을 수도 있다. 컨텍스트마다 다른 기술 스택(MVC+JPA vs Netty+MyBatis vs NoSQL)도 가능. ([#9](../sources/ddd-bounded-context.md))

## 컨텍스트 간 관계
- **고객/공급자(하류/상류)**: API를 호출하는 하류(downstream)가 제공하는 상류(upstream)에 의존. 상호 협력·일정 협의가 관건.
- **오픈 호스트 서비스**: 상류가 여러 하류 요구를 수용하는 단일 API로 공개해 일관성 유지.
- **안티코럽션 계층(ACL)**: 하류가 상류 모델이 자기 도메인을 침범하지 못하게 막는 보호 계층. 앞의 `RecSystemClient`가 외부 추천 모델을 카탈로그 모델로 변환하는 것이 예.
- **공유 커널(Shared Kernel)**: 두 팀이 한 모델을 공유. 중복은 줄지만 임의 변경이 서로에게 영향 → 밀접 협력 필수.
- **독립 방식**: 통합하지 않고 수동(사람이 직접 입력)으로. 규모가 커지면 한계. ([#9](../sources/ddd-bounded-context.md))

## 통합 방식: 직접 vs 간접
- **직접 통합**: REST API 직접 호출. 카탈로그가 관점의 도메인 서비스 인터페이스(`ProductRecommendationService`)를 정의하고, 구현(ACL)은 인프라에 두어 외부 모델을 변환한다 → [도메인 서비스](../entities/domain-service.md).
- **간접 통합**: **메시지 큐**. 비동기라 발신 측이 처리를 기다리지 않는다. **큐를 누가 제공하느냐가 메시지 구조(누구의 도메인 모델을 따를지)를 결정**한다. 큐 제공자가 출판하고 다른 컨텍스트가 구독하는 **출판/구독 모델**. ([#9](../sources/ddd-bounded-context.md))

## 이벤트로 강결합을 낮추기 (10장과의 연결)
동기 직접 호출로 컨텍스트를 잇는 강결합은 상대의 변경·성능·장애를 내 시스템에 전이시킨다. 10장은 이를 **도메인 이벤트, 특히 비동기 이벤트**로 낮춘다 → [도메인 이벤트](ddd-domain-events.md). 즉 컨텍스트 통합의 "간접 통합/출판·구독"은 이벤트 장의 메시징·이벤트 저장소 구현과 직접 이어진다. ([#10](../sources/ddd-domain-event.md))

## 마이크로서비스와 컨텍스트 맵
마이크로서비스는 컨텍스트별 모델 분리·독립 배포와 잘 어울린다(컨텍스트마다 프로젝트 분리). 전체를 조망하려면 **컨텍스트 맵**으로 경계·관계(OHS·ACL)·하위 도메인·조직을 한눈에 그린다. 정해진 규칙은 없고 이해가 깊어지면 맵도 바뀐다. ([#9](../sources/ddd-bounded-context.md))
