# 도메인 이벤트

이벤트는 트랜잭션 경계와 컨텍스트 경계를 넘는 변경을, 강결합 없이 처리하는 수단이다. 3장·8장에서 예고되고, 10장에서 본격 구현되며, 11장에서 CQRS의 저장소 동기화로 재등장한다.

## 문제: 동기 직접 호출의 강결합
한 도메인 기능 뒤에 외부 시스템 연동(결제·알림)을 **동기로 직접 호출**하면 (1) 트랜잭션 문제(외부 실패 시 롤백 여부 애매) (2) 성능 문제(외부가 느려지면 내 기능도 느려짐) (3) 설계 문제(도메인 객체에 외부 로직이 섞임)가 생긴다. 근본 원인은 **두 바운디드 컨텍스트의 강결합**이다 → [바운디드 컨텍스트와 통합](ddd-bounded-context-and-integration.md). ([#10](../sources/ddd-domain-event.md))

## 이벤트 개념과 구성요소
이벤트는 **과거에 벌어진 일**(상태 변경)이다. 요구사항의 "~할 때, ~하면"은 대개 상태 변경 신호다. 네 구성요소:
1. **이벤트** — 종류(클래스명)·발생 시간·최소한의 추가 데이터. 이름은 **과거 시제**(`Changed`, `Canceled`).
2. **이벤트 생성 주체** — 엔티티·밸류·도메인 서비스. `Order.changeShippingInfo()`가 `Events.raise()`로 발생.
3. **디스패처/퍼블리셔** — 생성 주체와 핸들러를 연결.
4. **핸들러/구독자** — 이벤트 데이터로 후속 작업.

이벤트에는 핸들러가 필요한 데이터만 담는다(무관한 데이터 배제). ([#10](../sources/ddd-domain-event.md))

## 효과: 의존 제거
주문 취소에 이벤트를 적용하면 `Order.cancel()`은 환불 서비스를 직접 호출하지 않고 `OrderCanceledEvent`만 발생시킨다. **주문 도메인 → 결제 도메인 의존이 사라지고**, 확장 시 핸들러만 추가하면 된다(기존 취소 로직 불변). 이것이 3장에서 "한 트랜잭션에 한 애그리거트"를 지키며 다른 애그리거트 변경을 유도하는 방법이다 → [애그리거트 설계](ddd-aggregate-design.md). ([#3](../sources/ddd-aggregate.md), [#10](../sources/ddd-domain-event.md))

## 스프링 구현
`ApplicationEventPublisher`를 디스패처로, `Events.raise()`(정적 헬퍼)로 발행, `@EventListener`로 수신. 기본은 **응용 서비스와 핸들러가 같은 트랜잭션**에서 실행된다. ([#10](../sources/ddd-domain-event.md))

## 동기의 한계와 비동기 처리
동기 핸들러는 외부 성능·트랜잭션 문제가 여전하다. "A 하면 B 하라"는 실제로는 **"A 하면 최대 언제까지 B 하라"**인 경우가 많아 비동기가 적합하다. 네 방법:
1. **로컬 핸들러 비동기**(`@Async`+`@EnableAsync`) — 간단하나 실패 시 유실 가능.
2. **메시징 시스템**(카프카·래빗MQ) — 보통 다른 프로세스. 안전 전달엔 글로벌 트랜잭션 필요(래빗MQ 지원, 카프카 미지원·고성능).
3. **이벤트 저장소 + 포워더** — DB에 저장(상태 변경과 같은 로컬 트랜잭션 가능), 포워더가 주기적으로 읽어 전달. 재처리 가능.
4. **이벤트 저장소 + 제공 API** — 외부 핸들러가 API로 이벤트를 가져가고 처리 위치(offset)를 스스로 기억.

`EventStore`는 이벤트가 불변이므로 **추가·조회만** 제공. 자동 증가 칼럼은 insert 시점과 커밋 시점의 시차로 누락이 생길 수 있어 지연 조회 고려. ([#10](../sources/ddd-domain-event.md))

## 추가 고려사항
소스 정보 저장 여부, 포워더 재시도 횟수 제한(계속 실패하는 이벤트가 뒤를 막음), 이벤트 손실(저장소 방식은 트랜잭션으로 보장, 로컬 비동기는 유실 가능), 순서 보장(저장소는 순서대로), **재처리와 멱등성**(마지막 처리 순번 기억 또는 핸들러를 멱등하게). ([#10](../sources/ddd-domain-event.md))

## 이벤트와 DB 트랜잭션
동기든 비동기든 이벤트 처리 실패와 트랜잭션 실패를 함께 고려해야 한다. 경우의 수를 줄이는 핵심은 **트랜잭션 성공 뒤에만 핸들러 실행** — 스프링 `@TransactionalEventListener(phase = AFTER_COMMIT)`. 단 이걸 붙인다고 모든 게 해결되진 않고(사용자 강조), 이후엔 이벤트 처리 실패만 재시도·실패 저장·수동·멱등으로 다룬다. ([#10](../sources/ddd-domain-event.md))

## CQRS 저장소 동기화와의 연결
명령 모델과 조회 모델이 다른 저장소를 쓸 때, 상태 변경 이벤트를 조회 모델에 전달해 동기화한다. 즉시 반영은 동기+글로벌 트랜잭션, 지연 허용은 비동기 → [CQRS와 조회 모델](ddd-cqrs-and-read-model.md). ([#11](../sources/ddd-cqrs.md))
