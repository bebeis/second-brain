# 바운디드 컨텍스트

## 모델은 문맥 안에서만 완전하다

도메인 모델을 만들 때 흔히 빠지는 함정이 **모든 하위 도메인을 하나의 완전한 모델로** 표현하려는 것이다. 하나의 도메인은 여러 하위 도메인으로 나뉘고, 각 하위 도메인은 같은 용어라도 다른 의미를 갖는다. 같은 "상품"이라도 어디선 이름·가격·설명 같은 **정보**를, 어디선 **개별 물건을 추적하는 대상**을 뜻한다. 반대로 같은 대상을 하위 도메인마다 다른 용어로 부르기도 한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image.png)<!-- {"width":581} -->

그래서 하나의 모델로 전부 표현하면 의미가 약해지고, 하위 도메인마다 다르게 발전하는 요구사항을 못 담는다. 제대로 된 모델은 **하위 도메인마다 따로 만들고 섞이지 않게 경계를 둬야** 한다. 결국 모델은 특정 문맥 안에서만 완전한 의미를 갖는데, 이렇게 경계를 가진 문맥을 DDD에서 **바운디드 컨텍스트(BC)**라 부른다.

## 바운디드 컨텍스트의 성질과 조직 현실

BC의 기본 성질은 이렇다 — 모델의 경계를 결정하고, 논리적으로 한 개의 모델을 가지며, 주로 용어를 기준으로 구분되고, 기능을 제공하는 물리적 시스템이다. 카탈로그 컨텍스트와 재고 컨텍스트는 용어·의미가 다르니 별도로 나눌 수 있다.

이상적으론 하위 도메인과 BC가 1:1이면 좋지만, 현실은 **조직 구조**에 좌우된다. 한 하위 도메인을 맡은 팀이 기능별로 갈리면 그 안에서 BC가 여러 개로 쪼개지고, 반대로 규모가 작아 한 팀이 전체를 구현하면 여러 하위 도메인이 하나의 BC에 들어간다. 후자에서도 중요한 건 **모델이 섞이지 않게 하는 것**이다. 한 프로젝트에 여러 하위 도메인이 들어가면 단일 모델의 유혹이 생기지만, 그러면 각 하위 도메인 특성을 못 살리고 확장이 어렵다. 그래서 **물리적으로 하나의 BC라도 패키지를 구분해 논리적 BC를 유지**해야 한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%203.png)<!-- {"width":650} -->

BC는 자신이 구현하는 하위 도메인에 맞는 모델을 담는다. 같은 사용자라도 회원 컨텍스트에선 `Member` 애그리거트 루트, 주문 컨텍스트에선 `Orderer` 밸류다. 같은 상품이라도 카탈로그의 `Product`는 `Category`와 관계를 맺지만 재고의 `Product`는 아니다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%204.png)<!-- {"width":541} -->

## 구현: 도메인 방식이 전부는 아니다

BC는 도메인 모델만이 아니라 기능 제공에 필요한 모든 요소를 포함한다. 모델 구조가 바뀌면 스키마도 바뀌니 **DB 테이블도 포함**되고, **표현 영역**은 HTML을 줄 수도 다른 컨텍스트를 위한 REST API를 줄 수도 있다.

**모든 BC를 도메인 주도로 개발할 필요는 없다.** 상품 리뷰처럼 복잡한 로직이 없으면 DAO와 데이터 중심 객체를 쓰는 CRUD로 충분하다. 서비스-DAO 구조는 도메인 기능이 서비스에 흩어질 수 있지만, 기능이 단순하면 문제되지 않는다. 한 BC 안에서 방식을 섞을 수도 있는데 대표적인 게 **CQRS** — 상태 변경은 도메인 모델 기반으로, 조회는 서비스-DAO와 조회 전용 모델로 구현한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%207.png)<!-- {"width":433} -->

BC마다 다른 기술도 가능하다. 어떤 건 스프링 MVC+JPA/하이버네이트, 어떤 건 Netty+MyBatis, 어떤 건 RDBMS 대신 MongoDB. UI가 꼭 있어야 하는 것도 아니다. 브라우저가 카탈로그에서 상품 상세를 받고 리뷰 컨텍스트의 REST API를 Ajax로 직접 호출할 수도 있고, 아니면 **UI 서버를 따로 둬서** 여러 BC의 정보를 읽어 조합하는 **파사드**로 쓸 수도 있다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%209.png)<!-- {"width":623} -->

## 컨텍스트 간 통합

온라인 쇼핑몰에 개인화 추천을 도입해 카탈로그 컨텍스트와 추천 컨텍스트가 함께 생겼다고 하자. 상품 상세 하단에 유사 상품을 보여주려면 통합이 필요한데, **두 컨텍스트의 모델이 다르다.** 카탈로그는 상품 중심 모델을, 추천 시스템은 추천 연산용 모델(상품 상세 없이 아이템 ID·추천 순위 위주)을 쓴다. 그래서 카탈로그는 추천 모델을 그대로 쓰면 안 되고, 받은 데이터를 **카탈로그 도메인 모델로 표현**해야 한다.

이를 위해 카탈로그 관점의 **도메인 서비스 인터페이스**를 둔다. `ProductRecommendationService`는 상품 ID를 받아 카탈로그의 `Product` 목록을 반환하고, **구현 클래스는 인프라스트럭처에** 두어 외부 추천 시스템과 연동하며 추천 모델을 카탈로그 모델로 바꾼다. 예컨대 `RecSystemClient`가 추천 REST API를 호출해 `itemId`·`type`·`rank` 응답을 받아 카탈로그의 `ProductId`·`Product`로 변환한다. 변환이 복잡하면 변환 전용 클래스를 따로 둔다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%2011.png)<!-- {"width":602} -->

통합 방식은 **직접 통합(REST API 직접 호출)**과 **간접 통합(대표적으로 메시지 큐)**으로 갈린다. 추천 시스템은 상품 조회·구매 같은 활동 이력이 필요한데, 카탈로그가 이 이력을 큐에 넣고 추천 컨텍스트가 읽어 계산에 쓴다. 큐는 비동기라 카탈로그는 넣은 뒤 기다리지 않고 자기 일을 한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%2014.png)<!-- {"width":615} -->

이때 **큐를 누가 제공하느냐가 메시지가 누구의 도메인 모델을 따를지 결정한다.** 카탈로그가 큐를 제공하면 `ViewLog`·`OrderLog` 같은 카탈로그 모델 메시지를 출판하고 추천 시스템이 구독해 자기 모델로 변환한다(**출판/구독 모델**). 반대로 추천 시스템이 큐를 제공하면 카탈로그가 `ActivityLog`(`itemId`·`userId`·`activityType`·`actionDate`)처럼 추천 관점 메시지를 넣는데, 이건 큐를 쓴다는 점만 빼면 추천 REST API에 데이터를 넘기는 것과 크게 다르지 않다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%2020.png)<!-- {"width":602} -->

이 관점은 **마이크로서비스와 잘 맞는다.** 컨텍스트를 마이크로서비스로 구현하면 프로젝트가 분리돼 모델이 자연스럽게 나뉘고, 별도 프로세스라 독립 배포·모니터링·확장이 쉽다.

## 컨텍스트 간 관계

BC는 어떤 식으로든 연결되는데, 가장 흔한 건 한쪽이 API를 제공하고 다른 쪽이 호출하는 관계(대표적으로 REST API)다. 이때 API를 쓰는 쪽이 제공하는 쪽에 의존한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%2021.png)<!-- {"width":541} -->

- **고객/공급자(하류/상류)**: 호출하는 카탈로그가 **하류(downstream)**, 제공하는 추천이 **상류(upstream)**. 상류가 API를 마음대로 바꾸면 하류가 끌려다니고, 반대로 상류 변경마다 하류 승인을 받아야 하면 상류가 새 시도를 못 한다. 그래서 두 팀은 개발 계획을 공유하고 프로토콜(REST, Protocol Buffers 등)을 협의한다.
- **오픈 호스트 서비스(OHS)**: 하류가 여럿이면 상류가 여러 요구를 수용하는 단일 API를 서비스로 공개해 일관성을 유지한다.
- **안티코럽션 계층(ACL)**: 상류 서비스는 상류의 도메인 모델을 따르므로, 하류는 그 모델이 자기 모델을 침범하지 않게 보호해야 한다. 앞의 `RecSystemClient`가 외부 모델을 내부 카탈로그 모델로 변환하는 게 바로 이 역할이다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%2023.png)<!-- {"width":638} -->

- **공유 커널(Shared Kernel)**: 같은 대상을 다루는 두 팀이 하나의 모델을 함께 쓴다. 중복을 줄이지만, 한 팀이 임의로 바꾸면 다른 팀에 영향을 줘 밀접한 협력이 필요하다. 유지 못 하면 지연·정체가 더 큰 문제가 된다.
- **독립 방식**: 통합하지 않고 각자 모델을 발전시키며 **통합은 수동으로** 한다(사람이 한쪽 데이터를 보고 다른 쪽에 입력). 항상 나쁜 건 아니지만 규모가 커지면 한계가 온다.

## 컨텍스트 맵

개별 BC에만 집중하면 전체를 못 본다. 전체 비즈니스를 조망하는 지도가 **컨텍스트 맵**이다. BC 간 경계와 관계를 한눈에 보여주고, 주요 애그리거트·OHS·ACL·하위 도메인·조직 구조를 함께 표시하면 관계가 더 명확해진다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%239%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%EA%B3%BC%20%EB%B0%94%EC%9A%B4%EB%94%94%EB%93%9C%20%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8/image%2025.png)<!-- {"width":520} -->

컨텍스트 맵으로는 하위 도메인과 안 맞는 BC가 있는지, 조정이 필요한지, 핵심 도메인에 조직 역량을 어디에 집중할지를 볼 수 있다. 그리는 정해진 규칙은 없고 화이트보드·파워포인트로도 충분하며, 이해가 깊어지거나 관계가 바뀌면 맵도 같이 바뀐다.
