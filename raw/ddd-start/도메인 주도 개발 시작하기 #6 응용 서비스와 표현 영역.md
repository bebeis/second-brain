# 응용 서비스와 표현 영역

도메인 영역을 잘 만들어도 그것만으로는 사용자의 요구를 채우지 못한다. 사용자와 도메인을 잇는 매개체가 필요한데, 그게 **표현 영역**과 **응용 영역**이다.

- **표현 영역**: HTTP 요청의 URL·파라미터·쿠키·헤더로 실행할 기능을 판단하고, 그 기능을 제공하는 응용 서비스를 호출한다. 결과는 HTML·JSON 등 요청에 맞는 형식으로 변환해 응답한다.
- **응용 영역**: 서비스가 실제 기능을 제공한다. 필요한 값을 인자로 받고 결과를 리턴한다.

핵심은 **응용 서비스가 표현 영역에 의존하지 않는 것**. 사용자가 브라우저·REST API·TCP 소켓 중 무엇을 쓰든 몰라도 되게, 입력을 받고 결과를 돌려주는 데만 집중하면 된다.

## 응용 서비스는 흐름만 제어한다

응용 서비스는 표현 영역과 도메인 영역을 잇는 창구이자 디자인 패턴상 **파사드**에 가깝다. 흐름은 단순하다 — 기존 애그리거트를 쓰는 기능은 (조회 → 없으면 예외 → 도메인 기능 실행 → 리턴), 새로 만드는 기능은 (검증 → 생성 → 저장 → 리턴).

**응용 서비스가 복잡하다면 도메인 로직이 새어 들어온 것**이다. 예컨대 암호 변경에서 "기존 암호 일치 확인"은 도메인 핵심 규칙이라 애그리거트에 있어야 한다. 응용 서비스가 직접 비교·세팅하면 (1) 데이터와 로직이 흩어져 응집도가 낮아지고 (2) 탈퇴 등 다른 기능에서 같은 코드를 중복 구현하게 된다. 도메인 로직을 도메인에 모으면 중복이 줄고 변경이 쉬워진다.

## 서비스 크기와 인터페이스

구성은 두 갈래다. 한 클래스에 한 도메인의 여러 기능을 몰아넣으면 공통 로직을 private로 빼기 쉽지만 클래스가 비대해지고 관련 적은 코드·의존이 섞인다. 기능별로 서비스를 나누면 각자 필요한 의존만 가져 품질 유지가 쉽지만 로직 중복이 생길 수 있다(공통 헬퍼로 해결). **나는 기능별로 나누는 쪽을 선호한다.**

> 책에선 다루진 않지만, 사실 중복되는 로직을 Service 아래에 넣으면 해결된다. 제민님의 지속 가능한 소프트웨어 아키텍처의 Implement Layer를 둠으로써 재사용성이 늘어나는 것이다.

**인터페이스는 대개 불필요**하다. 구현이 여럿이거나 런타임 교체가 필요할 때 쓸모 있는데 응용 서비스는 둘 다 드물다. 이유 없이 `...Service`/`...ServiceImpl`을 나누면 파일만 늘어난다(YAGNI). Mockito는 클래스 자체도 대역으로 만들 수 있어 단위 테스트에도 굳이 필요치 않다.

## 파라미터와 리턴, 표현 영역 비의존

메서드는 실행에 필요한 값을 파라미터로 받는다. 전달 값이 둘 이상이면 별도 요청 클래스로 묶는 편이 편하다(웹 프레임워크가 HTTP 파라미터를 객체로 변환해 준다). 결과는 표현 영역에 필요한 값만 리턴한다 — **애그리거트를 그대로 리턴하는 건 비권장**이다. 표현 영역이 도메인 기능을 실행할 여지가 생겨 로직이 분산되기 때문.

파라미터·리턴 타입에 `HttpServletRequest`·`HttpSession` 같은 표현 기술을 쓰면 안 된다. 그러면 (1) 단독 테스트가 어렵고 (2) 표현 구현 변경이 전파되며 (3) 응용 서비스가 표현 역할까지 떠맡는다. 세션을 응용 서비스가 만지면 표현 영역만 봐선 세션 변화를 추적할 수 없게 된다.

## 트랜잭션

상태 변경은 반드시 트랜잭션으로 묶는다. 가입 성공 화면을 봤는데 DB 저장이 안 되면 로그인이 안 되고, 배송지 변경 실패 화면을 봤는데 DB엔 반영되면 엉뚱한 주소로 배송된다. 스프링 `@Transactional`은 `RuntimeException`이면 롤백·아니면 커밋하므로 이 규칙만 따르면 트랜잭션 코드가 간결해진다.

## 표현 영역의 책임

세 가지 — (1) 화면 흐름 제어(폼 응답 → 전송 → 처리 → 결과 화면), (2) 요청을 응용 서비스 형식으로 변환하고 결과를 사용자 형식으로 제공(예외 → 에러 코드 → 뷰 메시지), (3) 쿠키·서버 세션으로 연결 상태 관리(권한 검사와 연결).

## 값 검증

원칙적으로는 응용 서비스에서 다 한다(필수·형식 같은 값 검증 + ID 중복 같은 논리 검증). 하지만 표현 영역은 오류를 모아 한 번에 보여줘야 한다. 응용 서비스가 오류마다 예외를 던지면 (1) 컨트롤러가 예외별 catch로 번잡해지고 (2) 첫 오류에서 멈춰 사용자가 폼을 여러 번 다시 입력하게 된다(UX 저하).

개선책은 두 가지다. 응용 서비스가 오류를 **하나의 목록으로 모아 단일 예외**로 던지고 표현 영역이 그걸 꺼내 `BindingResult`에 담거나, 아예 검증을 나눠 표현 영역이 필수·형식·범위를(스프링 `Validator`), 응용 서비스가 존재·중복 같은 논리 오류를 맡는다. 엄격함의 정도는 의견이 갈리는데, **요즘은 응용 서비스에서 형식·논리를 모두 처리하는 편** — 코드는 늘어도 응용 서비스 완성도가 올라간다.

## 권한 검사 — 세 지점

"사용자 U가 기능 F를 실행할 수 있는가"의 확인. 스프링 시큐리티 같은 프레임워크는 유연하지만 복잡해서, 이해가 부족하면 무작정 도입하기보다 시스템에 맞게 직접 구현하는 편이 유지보수에 나을 수 있다. 보통 세 곳에서 한다.

1. **표현 영역** — 주로 인증 여부. 서블릿 필터에서 인증 정보를 만들고 인증된 요청만 컨트롤러로 넘긴다. URL별 권한도 같은 방식(스프링 시큐리티도 필터 기반).
2. **응용 서비스** — URL만으로 안 되면 메서드 단위. 코드에 직접 넣지 않고 스프링 시큐리티 AOP `@PreAuthorize("hasRole('ADMIN')")`로 검사할 수 있다.
3. **도메인 객체 수준** — 게시글을 작성자 본인·관리자만 삭제 같은 경우. 본인 확인을 위해 애그리거트를 먼저 조회해야 하므로 URL·메서드 수준에서 못 잡고, 응용 서비스가 조회 후 권한 검사 서비스를 호출한다. 프레임워크 확장으로 통합할 수도 있지만 도메인마다 달라, 이해가 부족하면 직접 구현이 유리하다.

## 조회 전용 기능엔 응용 서비스가 없어도 된다

조회 전용 DAO를 호출만 하고 추가 로직·트랜잭션이 없다면(단일 쿼리 조회) 굳이 응용 서비스를 만들지 말고 **표현 영역이 DAO/리포지터리를 바로** 써도 된다. 늘 서비스를 만들던 습관에는 어색하지만, 기여가 없으면 만들지 않아도 된다. 조회 전용 기능은 CQRS에서 다시 이어진다.
