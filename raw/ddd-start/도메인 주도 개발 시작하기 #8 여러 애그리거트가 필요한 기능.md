# 애그리거트 트랜잭션과 잠금

> 원본 파일명은 "여러 애그리거트가 필요한 기능"이지만 실제 내용은 애그리거트 동시성·잠금이다.

## 동시 수정이 일관성을 깨는 이유

한 주문에 대해 운영자는 배송 상태를, 고객은 배송지를 동시에 바꾼다고 하자. 각 트랜잭션은 리포지터리로 **서로 다른 물리 객체**를 구한다(개념적으론 같은 주문이지만).

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%238%20%EC%97%AC%EB%9F%AC%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8%EA%B0%80%20%ED%95%84%EC%9A%94%ED%95%9C%20%EA%B8%B0%EB%8A%A5/image.png)<!-- {"width":385} -->

그래서 운영자가 상태를 바꿔도 고객 쪽 객체는 여전히 배송 전으로 보여 배송지를 바꿀 수 있다. 둘 다 커밋되면 DB엔 두 변경이 모두 반영되지만, **운영자가 옛 배송지 기준으로 상태를 바꾸는 사이 고객이 배송지를 바꿔 일관성이 깨진다.** 막으려면 (1) 운영자가 처리하는 동안 고객 수정을 막거나 (2) 고객이 먼저 바꿨으면 운영자가 다시 조회하고 수정하게 해야 한다. DBMS 트랜잭션만으론 부족해 애그리거트 단위 기법 — **선점 잠금**과 **비선점 잠금**(Pessimistic / Optimistic Lock)이 필요하다.

## 선점 잠금 (Pessimistic)

먼저 애그리거트를 구한 스레드가 끝낼 때까지 다른 스레드의 수정을 막는다. 스레드2는 스레드1이 커밋해 잠금이 풀릴 때까지 대기하고, 그제야 변경된 애그리거트를 본다. 주문 예시로는 운영자가 선점 잠금으로 구해 상태를 바꾸고 커밋하면, 대기하던 고객이 그 뒤 조회해 이미 배송 상태라 배송지 변경이 실패("이미 배송이 시작되어 변경할 수 없습니다").

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%238%20%EC%97%AC%EB%9F%AC%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8%EA%B0%80%20%ED%95%84%EC%9A%94%ED%95%9C%20%EA%B8%B0%EB%8A%A5/image%202.png)<!-- {"width":333} -->

구현은 보통 DBMS 행 잠금 — `for update` 같은 쿼리로 한 커넥션만 접근하게 한다. JPA는 `EntityManager.find()`에 `LockModeType.PESSIMISTIC_WRITE`(하이버네이트는 `for update`), 스프링 데이터 JPA는 리포지터리 메서드에 `@Lock(LockModeType.PESSIMISTIC_WRITE)`.

**교착 상태**를 조심해야 한다. 스레드1이 A를, 스레드2가 B를 잡은 뒤 서로 상대의 것을 잠그려 하면 둘 다 상대의 해제만 기다리며 멈춘다. 사용자가 많을수록 확률이 커지고 시스템이 마비될 수 있으니 **최대 대기 시간**을 준다 — JPA `javax.persistence.lock.timeout` 힌트(ms), 스프링 데이터 JPA `@QueryHints`. 단 힌트 적용 여부는 DBMS·JPA 구현체마다 다르니(쿼리별 vs 커넥션 단위) 확인이 필요하다.

## 비선점 잠금 (Optimistic)

선점 잠금은 **하나의 트랜잭션 안에서만** 유지되므로, 조회와 수정이 다른 트랜잭션인 경우를 못 막는다 — 운영자가 배송지를 조회한 뒤 상태 변경 전 사이에 고객이 배송지를 바꾸면, 운영자는 변경 전 주소로 배송을 준비하게 된다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%238%20%EC%97%AC%EB%9F%AC%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8%EA%B0%80%20%ED%95%84%EC%9A%94%ED%95%9C%20%EA%B8%B0%EB%8A%A5/image%206.png)<!-- {"width":556} -->

비선점 잠금은 동시 접근을 막지 않고 **반영 시점에** 변경 가능한지 확인한다. 애그리거트에 버전 값을 두고, 수정 쿼리는 "식별자 일치 AND 버전 일치"일 때만 실행되며 성공하면 버전을 1 올린다. 다른 트랜잭션이 먼저 수정해 버전이 바뀌면 이후 수정은 실패한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%238%20%EC%97%AC%EB%9F%AC%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8%EA%B0%80%20%ED%95%84%EC%9A%94%ED%95%9C%20%EA%B8%B0%EB%8A%A5/image%207.png)<!-- {"width":537} -->

JPA는 `@Version`으로 지원한다. 버전 필드에 붙이고 테이블에 버전 칼럼을 추가하면, JPA가 UPDATE에 버전 조건을 함께 걸고(버전 10이면 10일 때만 수정·11로 증가), 응용 서비스는 버전을 직접 다룰 필요 없이 도메인 기능만 실행하면 트랜잭션 종료 시점에 JPA가 잠금 쿼리를 낸다. 수정된 행이 0이면(누가 먼저 수정) 스프링 `@Transactional` 종료 시점에 `OptimisticLockingFailureException`이 나고, 표현 영역이 이를 잡아 충돌 안내를 보여준다.

**여러 트랜잭션으로 확장**하려면 화면에 버전을 함께 실어 보낸다. 조회 시 버전 A를 넘기고 수정 요청 시 A를 되보내면, 시스템은 현재 버전 B와 비교해 A≠B면 수정 불가, A=B면 수행한다(수행 중 다른 트랜잭션이 먼저 커밋하면 커밋 시점에 다시 충돌).

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%238%20%EC%97%AC%EB%9F%AC%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8%EA%B0%80%20%ED%95%84%EC%9A%94%ED%95%9C%20%EA%B8%B0%EB%8A%A5/image%208.png)<!-- {"width":525} -->

HTML 폼은 버전을 hidden input으로 전송하고 요청 DTO에 주문번호와 함께 담는다. 응용 서비스는 전달받은 버전과 현재 버전을 비교해 불일치면 `VersionConflictException`을 던진다. 결국 표현 영역은 두 예외를 처리할 수 있다 — 조회 이후 이미 수정된 경우(`VersionConflictException`)와 거의 동시 수정으로 커밋 시점 충돌(`OptimisticLockingFailureException`). 구분이 필요 없으면 응용 서비스에서 `OptimisticLockingFailureException`만 던져도 된다.

**강제 버전 증가**도 알아둘 것. 루트가 아닌 내부 엔티티만 바뀌면 루트 값이 그대로라 JPA가 루트 버전을 안 올릴 수 있다. 하지만 애그리거트 관점에선 내부가 하나라도 바뀌면 변경된 것이므로, `LockModeType.OPTIMISTIC_FORCE_INCREMENT`로 조회해 트랜잭션 종료 시 루트 버전을 강제로 올린다(스프링 데이터 JPA는 `@Lock`).

## 오프라인 선점 잠금

컨플루언스처럼 편집 중 안내만으로는 동시 수정을 완전히 막지 못한다. 수정 화면을 열어 둔 **여러 트랜잭션 구간 전체**에서 막으려면 오프라인 선점 잠금이 필요하다(선점 잠금은 한 트랜잭션 범위, 비선점 잠금은 나중 감지라 둘 다 이걸 못 한다). 수정 기능은 보통 (1) 폼을 보여주는 트랜잭션, (2) 데이터를 반영하는 트랜잭션으로 나뉘는데, **폼 요청 시 잠금을 선점하고 수정 요청 처리 후 해제**해 그 사이 다른 사용자의 폼 접근을 막는다.

폼을 열어 둔 채 프로그램을 종료하면 잠금이 안 풀리므로 **유효 시간**을 두고 지나면 자동 해제한다. 반대로 정상적으로 오래 편집 중인데 만료되면 곤란하니, 수정 폼에서 1분마다 Ajax로 유효 시간을 연장한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%238%20%EC%97%AC%EB%9F%AC%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8%EA%B0%80%20%ED%95%84%EC%9A%94%ED%95%9C%20%EA%B8%B0%EB%8A%A5/image%2010.png)<!-- {"width":567} -->

인터페이스는 `LockManager` — `tryLock(type, id) → LockId`(선점 시도), `checkLock(lockId)`(유효 확인), `releaseLock(lockId)`(해제), `extendLockExpiration(lockId, inc)`(연장). `type`·`id`가 잠글 대상을 식별하고(식별자 10인 Article이면 type=Article 구분값, id=10), 성공 시 받는 `LockId`는 확인·해제·연장에 쓰이므로 폼과 수정 요청 사이에 유지해야 한다.

흐름은 이렇다 — 폼 요청 시 `tryLock()`으로 잠금을 잡고 데이터+`LockId`를 반환 → 컨트롤러가 `LockId`를 뷰로 넘기고 폼은 hidden input으로 되전송 → 수정 요청 시 먼저 `checkLock()`으로 유효성을 확인한 뒤 기능 실행 → 끝나면 `releaseLock()`. 선점 실패 시 `LockException`을 던져 "다른 사용자가 수정 중, 나중에 재시도" 화면을 보여준다. **수정 요청 시 `checkLock()`을 먼저 하는 이유**는 (1) 유효 시간이 지나 다른 사용자가 이미 선점했을 수 있고 (2) 잠금을 선점하지 않은 사용자가 직접 수정 요청을 보낼 수 있어 막아야 하기 때문.

**DB 테이블로 구현**할 수 있다. 잠금 정보 테이블은 `type`·`id`·`lockid`·`expiration_time`을 갖는다. `(type, id)`를 기본키로 두면 같은 대상에 동시 잠금을 막고, `lockid`엔 유니크 인덱스를 준다(잠금마다 새 `LockId`).

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%238%20%EC%97%AC%EB%9F%AC%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8%EA%B0%80%20%ED%95%84%EC%9A%94%ED%95%9C%20%EA%B8%B0%EB%8A%A5/image%2011.png)<!-- {"width":556} -->

`LockData`는 `type`·`id`·`lockId`·`expirationTime`을 갖고 `isExpired()`로 만료를 판단한다. 구현 예시는 `JdbcTemplate` 기반 — `tryLock()`은 새 트랜잭션으로 실행되며 기존 유효 잠금이 있으면 `AlreadyLockedException`, 만료된 건 지운 뒤 새 `LockId`(UUID)를 삽입하고 실패/중복 키면 `LockingFailException`을 던진다(유효 시간은 현재 시간 + `lockTimeout`). 나머지 — `checkLock()`은 없거나 만료면 예외, `extendLockExpiration()`은 `inc`만큼 연장, `releaseLock()`은 삭제(작업 끝나면 반드시 호출).
