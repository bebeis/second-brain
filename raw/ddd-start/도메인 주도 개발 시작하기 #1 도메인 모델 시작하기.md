# 도메인 모델과 엔티티·밸류

## 도메인과 하위 도메인

소프트웨어로 해결하려는 문제 영역이 **도메인**이다. 온라인 서점이라면 사용자가 책을 검색·구매·결제하고 배송을 확인하는 그 전체가 도메인이고, 이건 다시 카탈로그·주문·혜택·결제·배송·회원·리뷰·정산 같은 **하위 도메인**으로 쪼개진다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%231%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0/image.png)<!-- {"width":304} -->

기억할 점 두 가지. 하나는 도메인 소프트웨어라고 **모든 걸 직접 구현하진 않는다**는 것 — 배송은 외부 업체 연동으로 추적만, 결제는 PG사에 맡기는 식이다. 다른 하나는 **하위 도메인 구성이 고정이 아니라는 것** — 작은 업체는 정산을 엑셀로 처리하고, 대형 장비 판매처는 카탈로그와 주문서 접수 정도만 필요할 수도 있다. 결국 구성은 비즈니스 상황을 따라간다.

## 전문가와 개발자의 지식 공유

각 영역엔 전문가가 있고 이들이 자기 경험을 근거로 기능을 요구한다. 개발자 작업의 첫 단추는 **요구사항 이해**인데, 여기서 어긋나면 원치 않는 기능·수정 비용·일정 지연으로 번진다.

그래서 가장 좋은 건 **개발자와 도메인 전문가의 직접 대화**다. 중간 전달자가 많을수록 정보가 왜곡된다. 단 전문가의 요구가 늘 정답은 아니다 — 그들은 소프트웨어 전문가가 아니라 기존 업무 방식에 맞춘 해결책을 말하기 쉽다. 표면적 요구를 그대로 받지 말고 "왜 필요한지, 진짜 풀려는 문제가 뭔지"를 확인해야 한다. **잘못된 요구사항은 잘못된 제품을 낳는다.**

## 도메인 모델이란

도메인 모델은 특정 도메인을 **개념적으로 표현**한 것이다. 주문 도메인을 객체로 그리면 Order가 나오고, Order는 주문번호·총액 같은 데이터와 배송지 변경·주문 취소 같은 기능을 함께 가진다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%231%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0/image%203.png)<!-- {"width":653} -->

도메인을 이해하려면 **주요 데이터와 기능을 함께** 봐야 하고, 객체 모델은 둘을 같이 담아서 모델링에 적합하다. 다만 꼭 객체일 필요는 없다 — 상태 변화가 핵심이면 상태 다이어그램, 관계가 핵심이면 그래프, 계산 규칙이 핵심이면 수식이 낫다. 중요한 건 표현 방식이 아니라 **도메인 이해에 도움이 되느냐**다.

이때의 도메인 모델은 이해를 위한 **개념 모델**이라, 그대로는 코드가 안 된다. DB·트랜잭션·성능을 고려한 **구현 모델**로 옮겨야 하고, 구현 모델은 개념 모델을 최대한 따르게 만든다. 그리고 **하위 도메인마다 같은 용어가 다른 의미**를 가진다 — 카탈로그의 "상품"은 판매 대상 정보지만 배송의 "상품"은 전달할 물리적 물건이다. 그래서 하위 도메인마다 모델을 따로 둔다.

## 도메인 모델 패턴 — 규칙을 객체 안에

여기서 말하는 "도메인 모델 패턴"은 앞의 개념 모델과 달리 **도메인 계층을 객체지향으로 구현하는 패턴**을 뜻한다. 핵심은 **업무 규칙을 도메인 객체 안에 두는 것**이다.

"출고 전에는 배송지 변경 가능", "배송 전에는 주문 취소 가능" 같은 규칙은 Order나 OrderState에 구현한다. 상태만으로 판단되면 OrderState에, 다른 정보도 필요하면 Order에 둔다. 규칙이 도메인 모델에 있어야 바뀔 때 변경 범위가 줄고 도메인 지식이 코드에 남는다.

개념 모델은 처음부터 완벽할 수 없다. 프로젝트가 진행되며 이해가 깊어지고 초기 모델은 계속 수정되므로, 처음엔 윤곽만 잡고 구현하며 점진적으로 키운다.

## 모델은 요구사항에서 점진적으로 나온다

도메인을 모른 채 코딩할 순 없다. 기획서·유스케이스·사용자 스토리·대화로 도메인을 이해하고 초기 모델을 만드는데, 모델링의 기본은 **핵심 구성요소·규칙·기능**을 찾는 일이다.

- 규칙은 모델에 반영한다. "주문은 상품을 최소 하나 포함"이면 Order 생성 시 목록이 비면 예외를 던진다. 배송지가 필수면 생성 시 없으면 예외.
- 상태에 따라 제약이 달라지는 규칙은 enum(PAYMENT_WAITING, PREPARING …)으로 상태를 표현한다.
- 이해가 깊어지면 메서드 이름도 다듬는다. 배송지 변경과 주문 취소가 "출고 전"이라는 같은 제약을 공유함을 알면 그 의미가 드러나게 이름을 고친다.

이렇게 만든 모델은 전문가·개발자와 공유하며 다듬어야 한다(화이트보드·위키가 유용). **문서화의 목적은 지식 공유**다. 상위 수준 문서는 전체 구조 파악에 좋고, 잘 쓰인 코드 자체도 도메인을 드러내는 문서가 된다.

## 엔티티 — 식별자로 구분한다

도메인 모델은 크게 엔티티와 밸류로 나뉘고, 이 둘을 제대로 구분해야 설계가 산다.

엔티티의 특징은 **식별자를 가진다**는 것. 주문번호는 주문마다 달라 Order의 식별자가 된다. 식별자는 생성부터 삭제까지 유지되므로 배송지나 상태가 바뀌어도 그대로다. 따라서 **식별자가 같으면 같은 엔티티**로 보고, equals/hashCode도 보통 식별자 기준으로 구현한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%231%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0/image%207.png)

식별자 생성 방식은 도메인·기술에 따라 다르다.

- **규칙 조합**: 주문번호·운송장번호처럼 날짜·시간·일련값 조합. 동시 생성 중복 주의.
- **생성기**: UUID, Nano ID 등. 특별한 규칙 없을 때.
- **직접 입력**: 회원 아이디·이메일. 중복 방지가 중요.
- **DB 일련번호**: 자동 증가 칼럼은 저장해야 값을 알 수 있어, 생성 시엔 식별자를 넘길 수 없고 저장 후 리포지터리가 채워준다.

## 밸류 — 개념적으로 완전한 하나

밸류 타입은 **개념적으로 완전한 하나**를 표현한다. receiverName과 receiverPhoneNumber가 따로 있으면 유추해야 하지만, Receiver 밸류로 묶으면 코드만 봐도 "받는 사람"이 드러난다. 주소도 Address로 묶으면 명확하다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%231%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0/image%208.png)<!-- {"width":392} -->

필드가 여럿일 필요는 없다. int인 price·amounts를 **Money**로 감싸면 "금액"이라는 의미가 살고, add·multiply 같은 기능을 붙이면 정수 연산이 아니라 돈 계산임이 드러난다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%231%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0/image%2013.png)<!-- {"width":277} -->

**밸류는 불변으로** 만드는 게 좋다. 값을 바꾸는 대신 새 객체를 만든다. Money가 가변이면 OrderLine 생성 뒤 외부에서 Money를 바꿔 내부 값이 틀어질 수 있는데, 불변이면 이런 문제도 없고 방어적 복사도 준다. 비교는 엔티티가 식별자로 하는 것과 달리 **모든 속성 값이 같은지**로 판단한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%231%20%EB%8F%84%EB%A9%94%EC%9D%B8%20%EB%AA%A8%EB%8D%B8%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0/image%2014.png)<!-- {"width":386} -->

식별자도 밸류로 만들 수 있다. 주문번호·회원 ID·이메일은 단순 문자열이 아니라 도메인 의미를 가지므로, String 대신 OrderNo·MemberId로 두면 명확하다. Order의 id 타입이 OrderNo면 이름이 id여도 주문번호임을 안다.

## 도메인 모델에 set을 넣지 않기

습관적인 public set은 도메인 의도를 흐린다. `changeShippingInfo`는 배송지 변경이 분명하지만 `setShippingInfo`는 값 설정만 뜻하고, `completePayment` 대신 `setOrderState`면 결제 완료라는 의미가 사라진다.

set은 객체를 **불완전한 상태**로 만들기도 한다. 기본 생성자로 만든 뒤 setter를 나눠 호출하면 주문자 없이 상태만 바뀐 이상한 객체가 나올 수 있다. 그래서 **생성 시점에 필요한 값을 생성자로 다 받고 거기서 필수·제약을 검사**한다. 내부용 private set은 괜찮지만 외부에서 상태를 바꾸는 public set은 피한다. 밸류는 특별한 이유가 없으면 불변으로 두고 set을 안 준다. (DTO는 로직이 없어 get/set이 있어도 영향이 작지만, 요즘 프레임워크는 private 필드에 직접 주입할 수 있어 불변 DTO도 가능하다.)

## 유비쿼터스 언어

코드에 도메인 용어를 반영하는 건 매우 중요하다. 주문 상태를 STEP1·STEP2로 두면 단계 의미를 따로 외워야 하고 `verifyStep1OrStep2` 같은 이름으론 규칙이 안 드러난다. 반면 PAYMENT_WAITING·PREPARING·SHIPPED처럼 쓰면 **코드가 곧 도메인 의미**가 되어 "출고 전"이 어느 상태인지 바로 읽히고 버그도 준다.

에릭 에반스가 강조한 **유비쿼터스 언어**가 이것이다 — 전문가·관계자·개발자가 함께 쓰는 공통 언어를 대화·문서·모델·코드·테스트에 일관되게 반영한다. 그래야 소통의 모호함이 줄고 개발자가 도메인과 코드를 불필요하게 번역하지 않는다. 이해가 깊어지면 더 나은 용어를 찾게 되는데, 새 용어는 코드·문서에도 반영해 최신 모델을 유지한다.

국내에선 도메인 용어의 영어 표현이 늘 고민이다(state vs status, kind vs type, 못 찾아서 gubun처럼 발음 표기). 하지만 도메인에 안 맞는 단어를 쓰면 코드가 점점 도메인에서 멀어지므로, 알맞은 표현을 찾는 노력을 아끼지 말아야 한다.
