# CQRS

## 단일 모델로 조회까지 감당할 때의 부담

조회 화면 하나를 만들 때 여러 애그리거트에 흩어진 데이터를 모아야 하는 경우가 흔한데, 상태 변경과 조회에 **같은 도메인 모델**을 쓰면 여기서 복잡도가 커진다. 참조 방식마다 걸리는 지점이 다르다.

- **식별자 참조**: JPA 즉시 로딩 같은 최적화를 쓰기 어렵고, 필요한 데이터를 한 번의 `SELECT`로 가져오기 힘들어 조회 성능이 문제된다.
- **객체 참조**: 같은 연관이라도 화면마다 즉시/지연 로딩이 갈리고, DBMS 전용 기능이 필요하면 네이티브 쿼리까지 가야 한다.

객체 지향 도메인 모델과 ORM은 상태 변경엔 잘 맞지만, 여러 애그리거트를 모아 출력하는 조회엔 고려할 게 많다. 그래서 복잡도를 낮추는 가장 단순한 답이 **상태 변경용 모델과 조회용 모델을 분리**하는 것이다. (책에는 이해를 돕는 참고 영상 안내도 함께 있다.)

## 명령 모델과 조회 모델을 나눈다

시스템 기능은 크게 **상태 변경**(주문 생성·배송지 변경·암호 변경)과 **상태 조회**(주문 상세·목록·통계)로 나뉘는데, 둘이 다루는 데이터 범위가 다르다. 상태 변경은 주로 한 애그리거트를, 조회는 여러 애그리거트를 함께 필요로 한다. 이 **범위 불일치** 때문에 한 모델로 둘 다 하면 불필요하게 복잡해진다. 이를 가르는 게 **CQRS(Command Query Responsibility Segregation)** — 상태를 바꾸는 **명령 모델**과 상태를 제공하는 **조회 모델**을 분리하는 패턴이다. 도메인이 복잡할수록 명령·조회의 데이터 범위 차가 커져 더 잘 맞는다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2311%20CQRS/image%202.png)<!-- {"width":568} -->

설계 초점부터 다르다. **명령 모델**은 상태 변경 도메인 로직에, **조회 모델**은 화면에 보여줄 데이터를 빠르게 읽는 데 초점을 둔다. 같은 기능이라도 명령 모델은 여러 도메인 객체로 구성되지만, 조회 모델은 화면에 필요한 데이터만 담은 **요약 타입 하나**로 끝낼 수 있다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2311%20CQRS/image%205.png)<!-- {"width":478} -->

**조회 모델엔 응용 서비스가 없어도 된다.** 단순 조회는 컨트롤러에서 바로 DAO를 실행해도 되고, 표현 영역에 넘길 때 별도 로직이 필요할 때만 응용 서비스를 둔다.

## 기술과 저장소 선택

각 모델에 맞는 기술을 고를 수 있다. 명령 모델은 객체 지향에 좋은 JPA로, 조회 모델은 SQL 조회에 강한 MyBatis로. 물론 꼭 다른 기술일 필요는 없고(5장 JPQL 동적 인스턴스, 하이버네이트 `@Subselect`도 조회 모델), JPA 하나로 둘 다 만들 수도 있다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2311%20CQRS/image%203.png)<!-- {"width":467} -->

저장소도 분리할 수 있다. 명령 모델은 트랜잭션을 지원하는 RDBMS를, **조회 모델은 조회 성능이 좋은 메모리 기반 NoSQL**을 쓰는 식이다. 이때 두 저장소 동기화는 10장의 이벤트로 처리한다 — 명령 모델에서 상태가 바뀌면 이벤트를 발생시켜 조회 모델에 반영한다.

- **즉시 반영**이 필요하면 동기 이벤트 + 글로벌 트랜잭션. 대신 응답 속도·처리량이 떨어질 수 있다.
- **특정 시간 안에만** 맞으면 되면 비동기로. 통계용 조회 저장소는 수초·수분·1시간 단위 반영으로 충분한 경우가 많아, 비동기 동기화로 명령 모델의 성능 저하를 줄인다.

## 웹은 사실상 CQRS로 수렴한다

웹은 변경보다 조회 요청이 훨씬 많다(게시글은 한 번 등록되지만 조회는 수백만 번). 그래서 쿼리 최적화, 화면 형태로 변환한 캐싱, 조회 전용 저장소 같은 성능 기법을 쓰게 되는데, 이건 결국 **조회 전용 모델을 쓰는 것과 같은 효과**다. 즉 대규모 트래픽 서비스는 명시하지 않아도 CQRS 비슷한 구조로 간다. 그럴 바엔 명령/조회를 명확히 구분하는 편이, 조회 때문에 명령 모델이 복잡해지는 걸 막고 조회 특화 기법도 자유롭게 적용할 수 있어 낫다.

## 도입은 트레이드오프

- **장점**: 명령 모델이 도메인 로직에 집중할 수 있고(조회 성능용 코드가 안 섞임), 조회 성능을 캐시·특화 쿼리·전용 저장소로 자유롭게 끌어올릴 수 있다.
- **단점**: 코드가 늘고, 두 모델·저장소·동기화용 메시징 등 더 많은 기술이 필요할 수 있다.

결국 단순한 도메인에 넣으면 두 모델 유지 비용만 커지고, 트래픽이 높은데 단일 모델을 고집하면 유지보수 비용이 커진다. 장단점을 비교해 결정할 문제다.
