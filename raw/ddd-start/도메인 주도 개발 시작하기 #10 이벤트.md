# 도메인 이벤트

## 강결합이 문제다

한 도메인 기능 뒤에 외부 시스템 연동(결제·알림 등)이 이어져야 할 때, 그걸 엔티티가 하든 응용 서비스가 하든 **외부를 동기로 직접 호출**하면 문제가 생긴다.

- **트랜잭션**: 외부 호출 중 예외가 나면 내 트랜잭션을 롤백할지, 커밋하고 외부 연동만 재시도할지 애매하다.
- **성능**: 외부 응답이 느려지면 내 기능도 같이 느려진다.
- **설계**: 도메인 객체에 외부 서비스가 섞이고, 외부가 바뀌면 내 도메인 객체도 영향받는다. 연동 대상이 늘수록 한 메서드에 여러 도메인 로직이 뒤섞인다.

근본 원인은 **두 바운디드 컨텍스트의 강결합**이다. 이걸 낮추는 방법이 **이벤트**, 특히 **비동기 이벤트**다.

## 이벤트란 무엇이고 무엇으로 구성되나

여기서 이벤트는 **과거에 벌어진 일**이다. 암호를 바꿨으면 "암호 변경됨", 주문을 취소했으면 "주문 취소됨" 이벤트가 발생한 것이고, 이는 보통 **상태가 바뀌었다**는 뜻이다. 요구사항에 "~할 때", "~가 발생하면" 같은 표현이 나오면 상태 변경과 관련된 경우가 많다. "주문을 취소할 때 이메일을 보낸다"면 "주문 취소됨 이벤트"를 발생시키고 핸들러에서 메일을 보내면 된다.

도입하려면 네 구성요소가 필요하다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image.png)<!-- {"width":604} -->

- **이벤트** — 종류(클래스 이름), 발생 시간, 처리에 필요한 추가 데이터를 담는다.
- **이벤트 생성 주체** — 엔티티·밸류·도메인 서비스. 상태가 바뀌면 관련 이벤트를 발생시킨다.
- **디스패처/퍼블리셔** — 생성 주체와 핸들러를 연결한다.
- **핸들러/구독자** — 이벤트에 담긴 데이터로 후속 작업을 한다(예: 주문 취소 이벤트를 받아 SMS 발송).

디스패처 구현에 따라 이벤트 생성·처리는 동기·비동기 어느 쪽으로도 갈 수 있다.

## 이벤트의 구성과 발생 흐름

이벤트 이름은 이미 벌어진 일이라 **과거 시제**(`Changed`, `Canceled`, `Created`)를 쓴다. 예컨대 배송지 변경은 `ShippingInfoChangedEvent`로, 주문번호·발생 시간·새 배송지를 담는다. 발생 주체는 `Order` 애그리거트이고, `Order.changeShippingInfo()`가 변경 가능 여부를 확인하고 배송지를 바꾼 뒤 `Events.raise()`로 이벤트를 낸다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%203.png)<!-- {"width":520} -->

이벤트에는 핸들러가 필요로 하는 데이터가 있어야 하고(부족하면 핸들러가 다시 조회해야 한다), 반대로 직접 관련 없는 데이터까지 담을 필요는 없다. 배송지 변경 이벤트엔 배송지만 담고 상품번호·개수 같은 건 넣지 않는다.

용도는 크게 둘이다.

- **트리거**: 상태 변경 뒤 후처리(주문 취소 → 환불, 예매 완료 → SMS)의 트리거로 쓴다.
- **시스템 간 데이터 동기화**: 배송지가 바뀌면 외부 배송 서비스에도 알린다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%205.png)<!-- {"width":540} -->

가장 큰 장점은 **서로 다른 도메인 로직의 섞임을 막는 것**이다. 주문 취소에 이벤트를 적용하면 `Order.cancel()`은 환불 서비스를 직접 부르지 않고 취소 상태로 바꾼 뒤 `OrderCanceledEvent`만 낸다. 환불 로직은 핸들러로 빠져 **주문 → 결제 의존이 사라진다.** 확장도 쉬워서, 취소 시 이메일 통지가 추가돼도 핸들러만 붙이면 되고 기존 취소 로직은 건드리지 않는다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%206.png)<!-- {"width":449} -->

## 스프링으로 구현하기

스프링 이벤트 기능을 쓰면 이벤트 클래스, 디스패처(`ApplicationEventPublisher`), 발행용 `Events` 클래스, 핸들러가 필요하다. 이벤트용 상위 타입이 꼭 필요하진 않고 원하는 클래스를 이벤트로 쓸 수 있지만, 공통 속성(예: 발생 시간)이 있으면 `Event` 추상 클래스를 만들어 생성 시점에 `timestamp`를 세팅하고 상속하게 할 수 있다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%208.png)<!-- {"width":548} -->

`Events`는 정적 필드로 퍼블리셔를 들고 `raise(event)`에서 `publishEvent(event)`를 부르는 정적 헬퍼다(도메인 어디서든 정적 메서드로 이벤트를 내려는 것). 정적 필드 초기화는 스프링 설정 클래스에서 `ApplicationContext`를 주입받아 넘긴다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2010.png)<!-- {"width":548} -->

핸들러는 `@EventListener(이벤트타입.class)`를 붙인 메서드다. `publishEvent()`에 이벤트를 넘기면 스프링이 그 타입을 처리하는 `@EventListener`를 찾아 실행한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2012.png)<!-- {"width":553} -->

전체 흐름은 응용 서비스 → 도메인 기능 → `Events.raise()` → `ApplicationEventPublisher` → `@EventListener` 실행이다. 이 기본 흐름에서는 **응용 서비스와 핸들러가 같은 트랜잭션 범위**에서 돈다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2014.png)<!-- {"width":610} -->

## 동기 처리의 한계

이벤트로 주문·결제 강결합은 줄지만, **핸들러가 동기면 외부 서비스 영향은 그대로 남는다.** 주문 취소 핸들러가 외부 환불 API를 부르는데 그게 느려지면 주문 취소도 느려지고, 환불에서 예외가 나면 주문 취소 트랜잭션을 롤백할지 고민이 남는다(실제론 취소는 처리하고 환불만 재처리·수동 처리해도 되는 경우가 많다). 결국 성능·트랜잭션 범위 문제라서, **비동기 처리**하거나 **이벤트 처리와 트랜잭션을 연계**해야 한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2015.png)<!-- {"width":483} -->

## 비동기로 처리하기

"A 하면 이어서 B 하라"는 요구는 실제론 **"A 하면 최대 언제까지 B 하라"인 경우가 많다.** 가입 후 인증 메일은 몇 초~수십 초 뒤여도 되고 못 받으면 재전송하면 되며, 주문 취소 후 결제 취소도 일정 시간 안이면 된다. 이런 건 "A 하면"을 이벤트로 보고 **비동기로 처리**한다. 대표 방법은 네 가지다.

**1. 로컬 핸들러 비동기 실행** — 가장 간단하다. 설정 클래스에 `@EnableAsync`, 핸들러에 `@Async`+`@EventListener`를 붙이면 스프링이 핸들러를 별도 스레드에서 돌린다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2016.png)<!-- {"width":443} -->

**2. 메시징 시스템** — 카프카·래빗MQ. 이벤트가 발생하면 디스패처가 큐에 보내고, 큐가 리스너에 전달해 핸들러를 실행한다. 저장 과정과 읽어 처리하는 과정은 별도 스레드·프로세스에서 돈다. 도메인 반영과 큐 저장을 한 트랜잭션으로 묶으려면 **글로벌 트랜잭션**이 필요한데, 안전하지만 성능이 떨어지고 미지원 시스템도 있다. (래빗MQ는 글로벌 트랜잭션·클러스터·고가용성 지원, 카프카는 글로벌 트랜잭션 미지원이지만 고성능.)

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2018.png)<!-- {"width":566} -->

**3·4. 이벤트 저장소 방식** — 이벤트를 먼저 DB에 저장하고 별도 프로그램이 읽어 전달한다. 도메인 상태와 이벤트 저장소가 같은 DB면 상태 변경·이벤트 저장을 하나의 로컬 트랜잭션으로 묶을 수 있고, 물리 저장이라 실패해도 재처리가 된다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2019.png)<!-- {"width":449} -->

- **포워더 방식**: 포워더가 주기적으로 저장소를 읽어 외부·핸들러에 전달하고 **처리 위치를 스스로 추적**한다.
- **API 방식**: 이벤트 목록을 주는 REST API를 두고 **외부 핸들러가 가져가며 자기 offset을 기억**한다.

저장소 구현엔 `EventEntry`(식별자·타입·payload(JSON)·발생 시간), `EventStore`(이벤트는 불변이라 `save`·`get(offset,limit)`만), `EventApi`(REST 컨트롤러), `EventStoreHandler`(`@EventListener(Event.class)`로 `Event` 상위 타입 이벤트만 저장)가 등장한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2021.png)<!-- {"width":604} -->

API 클라이언트는 마지막 처리 `lastOffset`을 저장해 두고(없으면 0), 그 offset부터 호출해 처리한 뒤 `offset + 읽은 개수`를 새 `lastOffset`으로 저장한다 — 같은 이벤트 중복 처리를 막기 위해서다. 이 방식은 장애 복구에도 좋아서, 실패하면 실패한 이벤트부터 다시 읽고 API 서버가 죽어도 복구되면 다시 가져온다. 포워더도 비슷하게 `EventStore`·`OffsetStore`·`EventSender`로 동작하며 `@Scheduled`로 주기 실행한다. 전송 중 예외가 나면 멈추고 다음 주기에 마지막 성공 이후부터 재시도한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2026.png)<!-- {"width":566} -->

> **저장소 기본키로 자동 증가 칼럼을 쓸 땐 주의.** 증가 값은 insert 시점에, 실제 반영은 커밋 시점에 일어난다. A가 id 11, B가 id 12를 받고 B가 먼저 커밋하면 두 커밋 사이에 조회 시 id 12는 보이고 id 11은 안 보이는 구멍이 생긴다. ID로 바로 다음 데이터를 읽지 말고 지연 조회 같은 방식을 고려한다.

## 적용 시 추가로 챙길 것

- **이벤트 소스 정보**: `EventEntry`는 발생 주체를 안 담아서 "Order가 낸 이벤트만 조회" 같은 게 안 된다. 필요하면 주체 정보를 추가한다.
- **포워더 전송 실패 허용 범위**: 특정 이벤트가 계속 실패하면 뒤 이벤트가 막히므로 **재시도 횟수를 제한**한다(예: 3회 실패 시 스킵). 버리지 말고 별도 실패 DB·큐에 남기면 원인 분석에 좋다.
- **이벤트 손실**: 이벤트 저장소 방식은 발생·저장을 한 트랜잭션으로 처리해 성공 시 저장을 보장하지만, 로컬 핸들러 비동기는 실패 시 유실될 수 있다.
- **순서**: 발생 순서대로 전달해야 하면 저장소가 낫다(순서대로 저장·제공). 메시징은 기술에 따라 순서가 달라질 수 있다.
- **재처리와 멱등성**: 마지막 처리 순번을 기억해 이미 처리한 순번은 무시하거나, 핸들러를 **멱등**하게 만든다. 배송지 변경 핸들러는 여러 번 처리해도 결과 주소가 같으니 멱등이고, 이러면 중복 발생·처리돼도 최종 상태가 같아 부담이 준다.

## 이벤트와 DB 트랜잭션

주문 취소와 환불을 이벤트로 구현한다고 하자. 동기든 비동기든 문제가 생길 수 있다 — 모두 동기면 환불 API까진 성공했는데 마지막 DB 업데이트가 실패해 "결제는 취소, DB는 미취소"가 되고, 비동기여도 주문 취소 커밋은 됐는데 이후 환불 API가 실패하면 "DB는 취소, 결제는 미취소"가 된다. 즉 **이벤트 처리 실패와 트랜잭션 실패를 함께** 봐야 한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%2310%20%EC%9D%B4%EB%B2%A4%ED%8A%B8/image%2030.png)<!-- {"width":460} -->

경우의 수를 줄이는 방법은 **트랜잭션이 성공한 뒤에만 핸들러를 실행**하는 것이다. 스프링 `@TransactionalEventListener(phase = AFTER_COMMIT)`은 커밋 성공 뒤에만 핸들러를 실행하고 롤백되면 실행하지 않는다. 이벤트 저장소를 DB로 쓰는 방식도 발생·저장을 한 트랜잭션으로 묶어 같은 효과를 낸다. 그러면 트랜잭션 실패 경우의 수가 줄고 **이벤트 처리 실패만 고려하면 된다.** (저거 붙인다고 모든 게 해결되는 건 아님 — 처리 실패는 여전히 남으니 재시도·실패 저장·수동 처리·멱등 중에 고른다.)
