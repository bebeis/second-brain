# DDD 아키텍처 — 네 영역과 DIP

## 네 개의 영역

DDD 아키텍처는 보통 **표현·응용·도메인·인프라스트럭처** 네 영역으로 나눈다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image.png)<!-- {"width":773} -->

- **표현**: 사용자 요청을 해석해 응용에 전달하고, 결과를 응답 형식으로 변환한다(스프링 MVC 컨트롤러). HTTP 파라미터를 응용 서비스용 객체로 바꾸고, 응용의 결과를 JSON·HTML로 바꾼다. 여기서 "사용자"는 브라우저일 수도 REST를 호출하는 외부 시스템일 수도 있다.
- **응용**: 사용자가 요청한 기능을 실행한다. 단 **도메인 로직을 직접 구현하지 않고** 도메인 객체를 구해 그 기능을 호출하며 흐름만 제어한다. 주문 취소면 주문을 찾아 `order.cancel()`을 부르는 식.
- **도메인**: 도메인 모델을 구현한다. Order·OrderLine·ShippingInfo가 여기 있고, 배송지 변경·결제 완료·총액 계산 같은 규칙을 담당한다.
- **인프라스트럭처**: RDBMS·메시징 큐·MongoDB·Redis·SMTP·REST 클라이언트 같은 **구현 기술 연동**. 나머지 영역은 이걸 직접 구현하지 않고 인프라가 제공하는 기능을 쓴다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%204.png)<!-- {"width":356} -->

## 계층 구조와 그 한계

네 영역을 흔히 계층 구조로 얹는다. 의존 방향은 표현 → 응용 → 도메인 → 인프라. 상위가 하위에 의존하고, 엄격히는 바로 아래에만 의존해야 하지만 편의상 응용이 인프라를 직접 쓰기도 한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%205.png)<!-- {"width":487} -->

문제는 **응용·도메인이 인프라의 구체 기술에 종속**된다는 점이다. 할인 금액 계산에서 응용 서비스가 룰 엔진을 직접 쓰면, (1) 그 기술·설정이 다 준비돼야 해서 응용만 따로 테스트하기 어렵고 (2) 코드가 그 기술의 데이터 구조·결과 타입에 간접적으로 묶여 구현 기술을 바꾸기 어렵다. 이걸 푸는 게 **DIP**다.

## DIP — 의존 역전 원칙

DIP의 핵심은 **고수준 모듈이 저수준 구현에 직접 의존하지 않게** 하는 것이다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%208.png)<!-- {"width":571} -->

할인 계산으로 보면, **고수준**은 "가격 할인 계산"이라는 의미 있는 기능을 제공하는 쪽이고 **저수준**은 고객 DB 조회·룰 엔진 실행처럼 실제 기술로 세부를 처리하는 쪽이다. 고수준이 저수준을 직접 쓰면 변경·테스트가 어렵다. 그래서 **고수준 관점의 필요 기능을 인터페이스로 추상화**한다 — "고객·주문 정보로 할인 금액을 계산한다"를 인터페이스로 두면, 고수준은 룰 엔진을 모른 채 인터페이스만 쓰고 실제 연동은 구현체가 맡는다. 그러면 의존 방향이 뒤집혀 **저수준이 고수준의 추상화에 의존**한다.

장점은 둘. 구현 기술을 바꿔도 고수준을 안 고치고, 실제 구현 없이 대역 객체(Mockito·람다)로 고수준을 테스트할 수 있다.

**주의** — DIP는 단순히 인터페이스/구현을 분리하는 게 아니다. 인터페이스를 저수준 관점에서 뽑으면 잘못 적용한 것이다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%2015.png)<!-- {"width":548} -->

`RuleEngine` 인터페이스를 인프라에 만들고 도메인이 그걸 의존하면 여전히 도메인이 인프라에 의존하는 셈이다. 올바른 추상화는 "룰 엔진 실행"이 아니라 "할인 금액 계산"처럼 **고수준의 의도**를 반영한다. 따라서 인터페이스는 저수준 기술이 아니라 고수준의 필요에서 도출하고, 위치도 고수준 쪽에 둔다.

아키텍처 수준에선 인프라가 저수준, 응용·도메인이 고수준이다. DIP를 적용하면 **인프라 클래스가 응용·도메인에 정의된 인터페이스를 구현**한다. 대표 예가 알림(응용에 인터페이스, 인프라에 이메일 구현)과 리포지터리(도메인에 `OrderRepository`, 인프라에 `JpaOrderRepository`). 단 DIP를 늘 완벽히 지킬 필욘 없다. 추상화 대상이 불분명하거나 DIP가 복잡도만 키우면 이점을 얻는 선에서 범위를 조절한다.

## 도메인 영역의 구성요소

도메인 영역의 주요 구성요소는 이렇다.

- **엔티티**: 고유 식별자와 자신만의 라이프 사이클을 갖는 객체. 데이터뿐 아니라 관련 기능도 제공한다.
- **밸류**: 식별자 없이 개념적으로 하나인 값(주소·금액·배송지). 엔티티나 다른 밸류의 속성으로 쓰인다.
- **애그리거트**: 관련 엔티티·밸류를 하나로 묶은 것. 주문 애그리거트는 Order·OrderLine·Orderer·ShippingInfo를 하나의 주문 개념으로 묶는다.
- **리포지터리**: 애그리거트를 저장소에 저장·조회.
- **도메인 서비스**: 특정 엔티티에 자연스럽게 안 붙는 도메인 로직(할인 계산처럼 상품·쿠폰·등급·금액 여러 모델이 필요한 것).

**도메인 모델의 엔티티는 DB 엔티티와 다르다.** DB 엔티티는 데이터 구조에 가깝지만 도메인 엔티티는 데이터와 기능을 함께 가진다. 그리고 두 개 이상의 데이터가 개념적으로 하나면 밸류로 표현한다 — Orderer가 이름·이메일을 묶어 주문자 개념을 드러내는 식. RDBMS 테이블은 칼럼으로 펼치거나 분리해야 해서 이런 밸류 개념을 잘 못 드러낸다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%2017.png)<!-- {"width":492} -->

밸류는 여기서도 불변이 좋다. 배송지를 바꿀 땐 기존 ShippingInfo를 고치지 않고 새 객체로 교체한다.

## 애그리거트와 루트

도메인이 커지면 엔티티·밸류가 많아져 개별 객체만으론 전체 구조를 못 본다. **애그리거트는 관련 객체를 묶어 상위 수준에서 모델을 보게** 한다 — 주문 애그리거트는 주문·배송지·주문자·주문 목록·총액을 하나로 묶어, 개별 객체가 아니라 애그리거트 간 관계로 도메인을 이해하게 해준다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%2020.png)<!-- {"width":563} -->

애그리거트엔 **루트 엔티티**가 있다. 루트는 속한 객체를 관리하고 애그리거트가 제공할 기능을 구현한다. 주문 애그리거트의 루트는 Order. 배송지를 바꿀 때 ShippingInfo를 직접 고치지 않고 `Order.changeShippingInfo()`를 써야 변경 가능 여부 같은 규칙을 항상 거친다. 애그리거트를 어떻게 구성하느냐에 따라 구현 복잡도·트랜잭션 범위·기술 제약이 달라진다(상세는 별도 정리).

## 리포지터리

도메인 객체를 계속 쓰려면 저장소에 보관해야 한다. 리포지터리는 **애그리거트 단위로 저장·조회**를 정의한다. 저장·조회 단위는 루트인 Order이고, `findByNumber`·`save`·`delete` 같은 메서드를 둔다. **인터페이스는 도메인 영역, 구현은 인프라**에 있다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%2021.png)<!-- {"width":587} -->

응용 서비스와 리포지터리는 밀접하다. 응용 서비스가 도메인 객체를 구하거나 저장할 때 리포지터리를 쓰고, 트랜잭션 처리는 리포지터리 구현 기술의 영향을 받는다.

## 요청 처리 흐름

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%2023.png)<!-- {"width":765} -->

사용자가 기능을 요청하면 표현 영역(컨트롤러)이 받아 형식을 검사하고, 문제없으면 응용 서비스용 형식으로 바꿔 위임한다. 응용 서비스는 리포지터리에서 도메인 객체를 가져와 기능을 실행하고 필요하면 새 객체를 만들어 저장한다(한 기능이 여러 도메인 객체를 쓸 수도 있다). **상태를 바꾸는 기능은 트랜잭션 안에서** 실행해야 하며, 스프링에선 보통 `@Transactional`을 쓴다.

## 인프라 의존은 무조건 없앨 필요 없다

DIP 관점에선 도메인·응용이 인프라를 직접 쓰기보다 인터페이스를 인프라가 구현하는 편이 유연하고 테스트하기 쉽다. 하지만 **구현 편의성도 중요한 가치**다 — 응용 서비스에 `@Transactional`을 붙이거나 도메인 모델에 `@Entity`·`@Table`을 쓰는 건 XML 매핑보다 간단하다. DIP의 장점을 크게 해치지 않는 선이면 일부 기술 의존을 둬도 괜찮고, 의존을 완전히 없애려다 설정·코드가 지나치게 복잡해지면 오히려 개발 비용만 커진다. (표현 영역은 원래 웹 기술과 밀접해서, 스프링 MVC냐 Vert.x냐에 따라 구현이 달라진다.)

## 모듈 구성

각 영역은 보통 별도 패키지에 둔다(`ui`·`application`·`domain`·`infrastructure`). 도메인이 크면 하위 도메인별(`catalog`·`order`·`member`)로 나누고 그 안에 네 패키지를 둔다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%232%20%EC%95%84%ED%82%A4%ED%85%8D%EC%B2%98%20%EA%B0%9C%EC%9A%94/image%2024.png)<!-- {"width":571} -->

더 세분화하려면 도메인 모듈을 애그리거트 기준으로 나누고(한 하위 도메인에 애그리거트가 여럿이면), 한 애그리거트의 모델과 리포지터리는 같은 패키지에 두며, 응용 서비스도 도메인별로 나눈다. 정답은 없고, 한 패키지에 타입이 너무 많아 찾기 불편하면(대략 10~15개 기준) 분리를 고려한다.
