# 리포지터리와 JPA 매핑

리포지터리는 애그리거트를 저장소에 넣고 다시 꺼내는 역할이다. RDBMS에서 객체 도메인 모델과 관계형 모델을 잇는 데는 ORM이 잘 맞고, 여기 정리는 자바 ORM 표준인 JPA를 기준으로 한다(JPA 세부 문법은 정리 대상이 아니다).

## 리포지터리의 위치와 기본기

인터페이스는 애그리거트와 함께 **도메인 영역**에, 구현 클래스는 **인프라스트럭처 영역**에 둔다. `domain.impl`에 둘 수도 있지만 인프라에 두는 편이 도메인의 인프라 의존을 낮춘다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image.png)<!-- {"width":507} -->

기본기는 두 가지 — **식별자로 조회**, **애그리거트 저장**. 인터페이스는 항상 **루트 기준**으로 잡는다(`Order`에 `OrderLine`·`Orderer`·`ShippingInfo`가 있어도 기준은 `Order`).

JPA 구현은 `EntityManager`를 쓴다. `find()`로 조회, `persist()`로 저장. **수정용 메서드는 대개 필요 없다** — 트랜잭션 안에서 조회 후 상태를 바꾸면 커밋 시점 변경 감지로 `UPDATE`가 나가기 때문. 식별자 외 조건 조회는 JPQL/Criteria로, 삭제는 `remove()`로 한다. 다만 실무는 조회·복구·보관 요구 때문에 바로 지우지 않고 **삭제 플래그로 노출만 제어**하는 경우가 많다.

## 애그리거트를 테이블에 매핑하기

기본 원칙은 단순하다. 루트는 엔티티이므로 `@Entity`, 한 테이블에 함께 저장되는 밸류는 `@Embeddable`(타입)/`@Embedded`(프로퍼티)로 매핑한다. 밸류가 밸류를 품어도 한 테이블에 들어갈 수 있고, 칼럼 이름이 다르면 `@AttributeOverride(s)`로 맞춘다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%202.png)<!-- {"width":543} -->

JPA는 기본 생성자를 요구하지만 도메인·밸류 객체는 생성 시점에 필요한 값을 다 받아야 하므로 public 기본 생성자는 두지 않고 **`protected`**로 둔다. 접근 방식은 **필드 접근**을 선호한다 — 메서드 접근은 getter/setter를 요구하는데, 공개 setter(`setState()`)보다 의미가 드러나는 메서드(`cancel()`, `changeShippingInfo()`)가 도메인을 더 잘 표현하기 때문.

밸류를 저장하는 방식은 상황별로 갈린다.

- **한 칼럼에 저장**: `AttributeConverter`로 밸류↔칼럼 타입을 변환. `@Converter(autoApply = true)`면 해당 타입에 자동 적용, 아니면 `@Convert`로 지정.
- **밸류 컬렉션을 별도 테이블에**: `@ElementCollection` + `@CollectionTable`, `List` 순서 보존은 `@OrderColumn`.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%206.png)<!-- {"width":516} -->

- **밸류 컬렉션을 한 칼럼에**: 컬렉션을 별도 밸류로 감싸 그 타입의 `AttributeConverter`로 직렬화해 저장.

**식별자도 밸류로** 만들 수 있다. `OrderNo`·`MemberId`는 단순 문자열보다 의미가 분명하고, 이때는 `@Id` 대신 `@EmbeddedId`를 쓴다. 단 JPA 식별자 타입은 `Serializable`을 구현하고 `equals()`/`hashCode()`도 제대로 만들어야 한다. 식별자를 밸류로 두면 **식별자 자체에 도메인 기능**(값 일부로 의미 구분 등)을 담을 수 있다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%2011.png)<!-- {"width":300} -->![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%2012.png)<!-- {"width":381} -->

**별도 테이블 = 엔티티**가 아니라는 점도 중요하다. 데이터가 별도 테이블로 분리돼도 독립된 식별성이 없고 루트의 일부일 뿐이면 밸류다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%2013.png)<!-- {"width":507} -->

이런 밸류는 `@SecondaryTable` + `@AttributeOverride`로 별도 테이블에 매핑한다. 단 `@SecondaryTable`은 조회 시 보조 테이블까지 조인하므로 목록 화면처럼 본문이 필요 없는 조회엔 비효율적 — 그럴 땐 조회 전용 모델을 따로 두는 게 낫다.

한편 **개념상 밸류인데 JPA 제약 때문에 `@Entity`로 매핑해야 할 때**도 있다. 밸류에 상속이 필요한 경우인데, JPA가 `@Embeddable` 상속을 지원하지 않아 `@Entity` 상속 매핑(`@Inheritance`, `@DiscriminatorColumn/Value`)을 쓴다. 그래도 **모델상 밸류라면 상태 변경 메서드는 두지 않는다.**

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%2016.png)<!-- {"width":566} -->

`@Entity`로 매핑한 밸류 컬렉션은 `@OneToMany`로 두고 루트와 라이프사이클을 함께하도록 `cascade`·`orphanRemoval`을 설정한다. 다만 `clear()` 시 하이버네이트가 각 엔티티를 개별 조회·삭제해 성능 문제가 생길 수 있어, 성능이 우선이면 상속을 포기하고 `@Embeddable` 단일 타입 + 타입 구분값으로 가기도 한다. **ID 참조 단방향 M-N**도 밸류 컬렉션과 비슷하게 식별자 집합을 `@ElementCollection`·`@CollectionTable`로 조인 테이블에 매핑하면 되고, 직접 참조가 없어 영속성 전파·로딩 고민이 줄어든다.

## 로딩 전략 — 완전한 애그리거트

핵심은 **루트를 조회하면 그에 속한 객체가 필요한 상태로 모여 하나의 완전한 애그리거트를 이뤄야 한다**는 것. 이를 즉시 로딩으로 보장할 수 있지만, 서로 다른 컬렉션을 함께 즉시 로딩하면 조인이 카타시안 곱처럼 불어나 성능이 나빠진다.

애그리거트가 완전해야 하는 이유는 (1) 상태 변경 실행, (2) 화면 표시인데, **화면 조회는 조회 전용 모델이 더 낫다.** 그래서 로딩 문제는 사실상 상태 변경 쪽 이야기다. 상태 변경은 트랜잭션 안 지연 로딩으로 필요한 시점에 필요한 것만 읽어도 되고, 조회가 상태 변경보다 훨씬 잦으니 상태 변경 시 추가 쿼리는 대개 문제가 안 된다. 결론은 **무조건 즉시/무조건 지연은 금물** — 크기·빈도·조회 방식·성능을 보고 고른다.

## 영속성 전파

애그리거트는 저장·삭제도 하나로 다뤄야 한다. `@Embeddable` 밸류는 루트와 함께 저장·삭제되니 별도 설정이 없다. 반면 내부 구성요소를 `@Entity`로 매핑했다면 `@OneToOne`·`@OneToMany`엔 기본 cascade가 없으므로 `CascadeType.PERSIST/REMOVE`를 명시하고, 컬렉션에서 빠진 객체를 DB에서도 지우려면 `orphanRemoval = true`를 준다.

## 식별자 생성

생성 방식은 셋 — 사용자 직접 입력 / 도메인 규칙 / DB 일련번호. 이메일처럼 사용자가 넣으면 도메인에 생성 기능이 필요 없다. **생성 규칙이 도메인에 속하면 생성 기능도 도메인 영역에** 둔다(생성 전담 도메인 서비스를 만들고, 응용 서비스가 그걸로 식별자를 얻어 엔티티를 만든 뒤 저장). 리포지터리에 `nextId()`를 두는 방법도 있다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%2018.png)<!-- {"width":471} -->

DB 자동 증가 칼럼은 `@GeneratedValue`. 이 방식은 insert가 나가야 식별자가 만들어지므로 **객체 생성 시점엔 식별자를 알 수 없고 저장 후에야** 쓸 수 있다(JPA의 다른 식별자 생성도 기본적으로 저장 시점 생성).

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%234%20%EB%A6%AC%ED%8F%AC%EC%A7%80%ED%84%B0%EB%A6%AC%EC%99%80%20%EB%AA%A8%EB%8D%B8%20%EA%B5%AC%ED%98%84/image%2020.png)<!-- {"width":529} -->

## 도메인 구현과 DIP — 실용적 타협

위 구현은 엄밀히 DIP를 완벽히 지키진 않는다. 도메인 모델에 `@Entity`·`@Table`·`@Id`·`@Column`이 붙고, 도메인 리포지터리가 스프링 데이터 JPA `Repository`를 상속하면 **도메인 패키지의 인터페이스가 인프라 기술에 의존**하기 때문이다. 완전히 끊으려면 도메인에서 JPA 애너테이션을 빼고 인프라에 JPA용 엔티티·리포지터리를 따로 둬야 한다.

하지만 **이 구현 기술은 실제로 잘 안 바뀐다**(JPA→마이바티스, RDBMS→몽고DB는 드묾). 애너테이션을 도메인에 써도 단위 테스트에 큰 지장이 없고, 스프링 데이터 JPA 리포지터리도 인터페이스라 테스트 가능성을 크게 해치지 않는다. 그래서 이 부분은 의도적으로 **실용적 타협**을 한 것 — 완벽한 DIP가 이상적이어도 구현 편의성과 복잡도 역시 가치이고, 애너테이션을 쓰면서도 구조적 유연성을 어느 정도 유지할 수 있다면 합리적인 선택이다.
