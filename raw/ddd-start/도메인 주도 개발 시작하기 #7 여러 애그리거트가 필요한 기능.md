# 도메인 서비스

## 한 애그리거트에 담기 애매한 로직

한 애그리거트만으로 안 풀리는 기능이 있다. 대표적인 게 **결제 금액 계산** — 상품 가격, 구매 개수, 쿠폰 할인, 회원 등급 할인 등 여러 애그리거트에 흩어진 정보가 필요하다. 그럼 계산 책임을 어디에 둘지 고민하게 되는데, 주문이 총 주문 금액은 계산해도 할인까지 반영한 결제 금액은 단순치 않고, 쿠폰이 계산하자니 **쿠폰을 여러 개 적용할 수 있으면 단일 쿠폰만으로는 전체를 못 구한다.**

주문 애그리거트에 데이터와 계산 책임을 다 몰아넣을 수도 있지만, 주문과 무관한 할인 정책(한시적 전 품목 추가 할인 등)이 생기면 **계산 책임이 거기 있다는 이유만으로 주문을 수정**해야 한다. 이렇게 억지로 넣으면 애그리거트가 책임 범위를 넘고 외부 의존이 늘어 복잡해지며, 무엇보다 **애그리거트 바깥의 도메인 개념이 안에 숨어 드러나지 않는다.** 이를 푸는 방법이 그 로직을 **별도 서비스로** 빼는 것이다.

## 도메인 서비스란

도메인 영역에 위치해 도메인 로직을 표현한다. 주로 두 경우 — (1) 여러 애그리거트가 필요하거나 한 애그리거트에 넣기 복잡한 **계산 로직**, (2) 구현에 다른 시스템이 필요한 **외부 연동 도메인 로직**. 응용 서비스가 응용 로직을 다룬다면 도메인 서비스는 도메인 로직을 다룬다.

애그리거트·밸류와 다른 점은 **상태 없이 로직만** 구현한다는 것. 필요한 상태는 파라미터로 받는다. 예컨대 할인 계산은 `DiscountCalculationService`로 표현하고, 주문 항목·쿠폰·회원 등급을 받아 할인 금액을 반환한다. 이름은 도메인 의미가 드러나게 짓는다.

## 사용 주체와 주입하지 않기

사용 주체는 둘이다. **애그리거트**가 메서드 파라미터로 도메인 서비스를 받아 계산하게 하거나, **응용 서비스**가 애그리거트를 준비한 뒤 도메인 서비스를 넘겨 실행하게 한다.

중요한 건 **도메인 서비스를 애그리거트에 필드로 주입하지 않는 것**. 스프링 DI에 익숙하면 필드로 넣고 싶어지지만, `DiscountCalculationService` 같은 필드는 애그리거트의 데이터가 아니고 DB 저장 대상도 아니며 모든 기능이 그걸 필요로 하지도 않는다. 일부 기능을 위해 필드 주입하는 건 적절치 않다. 실행 시 파라미터로 전달한다는 것 자체가 애그리거트가 도메인 서비스에 의존한다는 뜻이다.

반대로 **도메인 서비스 메서드에 애그리거트를 넘기는** 방식도 있다. 계좌 이체가 대표적 — 출금·입금 두 계좌가 함께 필요하니 도메인 서비스가 둘을 받아 처리하고, 응용 서비스가 조회 후 호출한다. 이때도 도메인 서비스는 **도메인 로직만** 하고 트랜잭션 같은 응용 로직은 응용 서비스가 맡는다.

응용/도메인 서비스가 헷갈리면 그 로직이 **애그리거트 상태를 변경/계산하는 도메인 로직인지**를 보면 된다. 계좌 이체는 상태를 바꾸고 결제 금액 계산은 값을 구하는데, 둘 다 도메인 로직이고 한 애그리거트에 담기 어려우니 도메인 서비스로 간다.

## 외부 시스템 연동

외부·타 도메인 연동도 도메인 서비스가 된다. 설문 시스템과 역할 관리 시스템이 분리돼 있어 설문 생성 시 권한 역할을 확인하려면 역할 시스템과 연동해야 하는데, 기술적으로는 HTTP API 호출이어도 설문 도메인 입장에선 "**사용자가 설문 생성 권한을 가졌는지 확인한다**"는 도메인 로직이다. 그래서 `SurveyPermissionChecker` 같은 인터페이스로 표현하되, **"역할 시스템과 연동한다"가 아니라 "생성 권한을 확인한다"는 도메인 로직 관점**으로 이름 짓는다. 응용 서비스가 이걸로 권한을 검사하고, 실제 구현 클래스는 인프라스트럭처에 두어 외부 시스템과 연동한다.

## 패키지 위치와 인터페이스

도메인 서비스는 도메인 로직을 표현하니 **도메인 영역**에 둔다(주문 금액 계산 서비스는 주문 애그리거트와 같은 패키지). 서비스가 많거나 엔티티·밸류·리포지터리와 구분하고 싶으면 `domain.model`·`domain.service`·`domain.repository`로 하위 패키지를 나눈다.

로직이 고정되어 있지 않거나 **외부 시스템·별도 엔진으로 구현**할 때는 인터페이스와 구현을 분리한다. 할인 계산을 룰 엔진으로 한다면 도메인 영역엔 `DiscountCalculationService` 인터페이스를, 룰 엔진 구현체(`RuleBasedDiscountCalculationService`)는 인프라스트럭처에 둔다.
