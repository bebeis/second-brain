# 애그리거트 — 경계·루트·참조

## 애그리거트가 필요한 이유

도메인 모델이 커지면 상품·주문·회원 같은 개념이 복잡하게 얽혀, 개별 객체 수준에서는 전체 관계를 파악하기 어렵고 변경할 때 큰 구조를 망가뜨리기 쉽다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image.png)<!-- {"width":425} -->

이걸 줄이는 개념이 **애그리거트**다. 관련 객체를 하나의 군집으로 묶어 상위 수준에서 도메인을 보게 한다. 주문 애그리거트는 주문·주문자·배송지·주문 목록·결제 정보를 포함하고, 그러면 애그리거트 간 관계로 모델을 이해할 수 있어 복잡도와 변경 부담이 준다.

애그리거트는 **일관성을 관리하는 기준**이기도 하다. 같은 애그리거트의 객체들은 보통 함께 생성·제거되며 유사한 라이프 사이클을 가지고(주문을 만들며 Order만 만들고 OrderLine·Orderer를 안 만드는 경우는 드물다), 경계가 있어 한 애그리거트 객체는 다른 애그리거트에 속하지 않는다(주문 애그리거트는 배송지·상품 개수는 관리해도 회원 비밀번호나 상품 가격은 안 바꾼다).

**경계는 도메인 규칙과 요구사항으로 정한다.** 함께 생성·변경되는 빈도가 높으면 같은 애그리거트일 가능성이 크지만, "A가 B를 가진다"만으로 판단하면 안 된다. 상품과 리뷰는 함께 보여도 생성 시점·변경 주체·라이프 사이클이 달라 서로 다른 애그리거트다. 처음엔 커 보여도 규칙을 이해할수록 실제 크기는 작아지는 경우가 많고, 엔티티 하나짜리 애그리거트도 흔하다.

## 애그리거트 루트와 일관성

애그리거트는 여러 객체로 구성되므로 **전체가 일관된 상태**를 유지해야 한다. 상품 수량이 바뀌면 OrderLine의 quantity뿐 아니라 Order의 totalAmounts도 함께 바뀌어야 "총액 = 각 상품 금액의 합"이 안 깨진다. 이 일관성을 책임지는 게 **애그리거트 루트**다. 루트는 애그리거트를 대표하는 엔티티이고 나머지는 루트에 직·간접으로 속한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image%204.png)<!-- {"width":377} -->

루트의 핵심 역할은 **일관성이 깨지지 않게 도메인 기능을 제공**하는 것이다. 배송 시작 전까지만 배송지 변경이 가능하면 `Order.changeShippingInfo()`가 먼저 변경 가능 여부를 검사한 뒤 바꾼다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image%205.png)<!-- {"width":495} -->

문제는 **외부에서 내부 객체를 직접 바꾸는 것**이다. `order.getShippingInfo().setAddress(...)`처럼 루트를 우회하면 상태 검사 규칙이 안 걸리고, 그 검사를 응용 서비스에 넣으면 같은 로직이 여기저기 중복된다. 이를 막는 두 습관 —

1. 단순 필드 변경용 public set을 만들지 않는다. `cancel()`·`changePassword()`·`changeShippingInfo()`처럼 의미가 드러나는 메서드가 도메인 규칙을 코드에 모은다.
2. 밸류를 불변으로 만든다. 배송지를 바꾸려면 새 ShippingInfo를 만들어 루트 메서드에 넘겨야 하고, 그러면 루트가 규칙을 적용한 뒤 상태를 바꾼다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image%206.png)<!-- {"width":534} -->

루트는 **내부 객체를 조합해 기능을 완성**한다. Order는 OrderLine 목록으로 총액을 계산하고, Member는 Password 객체로 기존 암호를 확인한 뒤 바꾼다. 루트가 늘 직접 계산할 필욘 없고 OrderLines 같은 별도 클래스에 위임할 수도 있지만, 외부에서 OrderLines를 직접 바꾸게 하면 totalAmounts와 어긋난다. 그래서 **내부 객체를 노출해도 변경 기능은 노출하지 않는다** — 불변으로 만들거나, 변경 메서드 접근 범위를 패키지·protected로 제한한다.

## 트랜잭션 범위는 작게

트랜잭션 범위는 작을수록 좋다. 한 트랜잭션에서 수정하는 테이블·행이 많을수록 잠금 대상이 늘고 처리량이 떨어진다. 같은 이유로 **한 트랜잭션에서는 가급적 하나의 애그리거트만 수정**한다. 한 애그리거트가 다른 애그리거트 상태까지 바꾸면 결합도가 높아진다 — 주문 배송지를 바꾸며 회원 주소까지 바꿔야 해도 Order 내부에서 Member를 직접 수정하면 안 되고, 부득이하면 응용 서비스에서 각 애그리거트를 조회해 각각의 기능을 호출한다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image%2010.png)<!-- {"width":645} -->

**도메인 이벤트**를 쓰면 한 트랜잭션에서 한 애그리거트만 수정하면서도 동기·비동기로 다른 애그리거트의 변경을 유도할 수 있다(별도 정리). 다만 팀 표준·기술 제약·UI 편의성 때문에 한 트랜잭션에서 여러 애그리거트를 수정해야 할 때도 있다.

## 리포지터리도 애그리거트 단위

애그리거트는 개념적으로 완전한 하나의 도메인 모델이므로 **리포지터리도 애그리거트 단위**로 만든다. Order와 OrderLine이 다른 테이블에 저장돼도 OrderLine용 리포지터리는 안 두고 루트인 Order 기준으로 OrderRepository만 둔다. 기본 기능은 `save`와 `findById`, 필요하면 조건 검색·삭제를 추가한다.

저장·조회는 애그리거트 전체가 대상이다. 저장 시 여러 테이블에 매핑됐다면 루트뿐 아니라 모든 구성요소가 저장돼야 하고, 조회된 Order는 기능 실행에 필요한 OrderLine·Orderer를 포함한 완전한 상태여야 한다(아니면 NPE 등이 난다). **변경은 원자적으로** 반영돼야 하며, RDBMS는 트랜잭션으로, MongoDB 같은 NoSQL은 한 애그리거트를 한 문서에 저장해 손실을 막는다.

## ID로 애그리거트 참조하기

애그리거트도 다른 애그리거트를 참조하는데, 관리 주체가 루트이므로 이는 곧 루트 참조다. 객체로 직접 참조하면 구현은 편하다(`order.getOrderer().getMember().getId()`, JPA `@ManyToOne` 지원). 하지만 문제가 셋 —

1. **결합도 상승**: 탐색이 편해 다른 애그리거트 상태를 바꾸려는 유혹이 생긴다.
2. **성능 고민**: 기능별로 지연·즉시 로딩을 판단해야 하고 조회 방식에 따라 JPQL·Criteria도 달라진다.
3. **확장 어려움**: 하위 도메인별로 저장소를 분리하면(RDBMS·MongoDB) JPA 객체 참조를 그대로 쓰기 어렵다.

그래서 **ID 참조** — 루트를 직접 참조하지 않고 식별자만 보관한다(Orderer가 Member 대신 MemberId). 경계가 명확해지고 객체 그래프가 과하게 얽히지 않아 복잡도가 낮아지며, 내부에서 다른 애그리거트 상태를 못 바꿔 응집도가 높아진다(필요하면 응용 서비스에서 ID로 조회). 저장소 확장에도 유리하다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image%2011.png)<!-- {"width":504} -->

단 ID 참조는 **조회 성능** 문제가 생길 수 있다. 목록 화면에서 여러 애그리거트를 함께 보여줘야 하면 항목마다 참조 애그리거트를 읽는 쿼리가 반복되는 **N+1 문제**다. 여기서 객체 참조·즉시 로딩으로 돌아가면 ID 참조의 장점을 잃으니, 대신 **조회 전용 쿼리**를 쓴다 — 조회 전용 DAO로 조인해 화면 데이터를 한 번에 가져온다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image%2013.png)<!-- {"width":563} -->

저장소가 다르면 조인으로 한 번에 못 가져오니 캐시나 조회 전용 저장소를 따로 구성한다. 코드는 복잡해지지만 트래픽 큰 시스템의 조회 성능엔 효과적이다.

## 집합 연관은 컬렉션 대신 ID로

애그리거트 간엔 1-N, M-N 연관이 있다(카테고리와 상품). **개념적 1-N이라고 반드시 컬렉션으로 구현할 필욘 없다** — 1쪽이 N쪽 목록을 컬렉션으로 들면 페이징 시 전체를 읽고 잘라내야 해서 성능 문제가 난다. 대신 **N쪽이 상대 ID를 갖게 하고** 그 ID로 목록을 조회한다. M-N도 양방향 컬렉션이 다 필요한 경우는 드물어, 보통 한쪽이 상대 ID 목록을 갖는 단방향으로 충분하다(JPA `@ElementCollection`으로 ID 집합 매핑, JPQL `member of`로 조회). 목록·상세 조회는 조회 전용 모델로 구현하는 게 낫다(별도 정리).

## 애그리거트를 팩토리로

한 애그리거트의 데이터로 다른 애그리거트를 생성할 때, 애그리거트를 **팩토리**로 쓸 수 있다. 예로 차단된 상점은 상품을 못 올린다고 하자. 응용 서비스에서 Store를 조회해 차단 여부를 검사하고 Product를 만들 수도 있지만, 그러면 "차단된 상점은 상품을 못 만든다"는 중요한 도메인 로직이 응용 서비스에 새어 나온다.

![](%EB%8F%84%EB%A9%94%EC%9D%B8%20%EC%A3%BC%EB%8F%84%20%EA%B0%9C%EB%B0%9C%20%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0%20%233%20%EC%95%A0%EA%B7%B8%EB%A6%AC%EA%B1%B0%ED%8A%B8/image%2015.png)<!-- {"width":632} -->

이 로직은 **Store에 두는 게 자연스럽다**. Store에 `createProduct()` 팩토리 메서드를 두고 그 안에서 차단 여부를 검사한 뒤 Product를 만들면, 응용 서비스는 상태를 직접 확인하지 않고 생성 기능만 호출한다. 장점은 응집도 — 생성 가능 규칙이 바뀌어도 응용 서비스가 아니라 Store만 고치면 된다. 애그리거트가 생성에 필요한 데이터를 갖고 생성 가능 여부도 자기 상태로 판단한다면 팩토리 메서드를 애그리거트에 두는 걸 고려한다. 정보가 너무 많으면 별도 팩토리에 위임하되, 차단 상태에 관한 도메인 로직은 여전히 Store 한 곳에 둔다.
